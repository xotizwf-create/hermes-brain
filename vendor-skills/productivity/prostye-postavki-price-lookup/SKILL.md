# Простые поставки — сверка по прайс-листу

Используй этот skill, когда Александр присылает скриншот, строку спецификации, товарное наименование или спрашивает «что это», «краткое наименование», «как это называется по прайсу» для проекта «Простые поставки».

## Главный принцип

Источник истины — **загруженный в коннектор прайс-лист**, таблица `price_list_entries`, а не память, прошлые ответы, внешняя логика или примерное совпадение по одному компоненту.

Поддерживай поиск в обе стороны:

- **длинное/полное наименование → краткое наименование/код**: пользователь присылает длинную строку из спецификации, договора, скриншота или Excel; нужно найти наиболее совпадающую строку прайса и вернуть краткое наименование/код;
- **краткое наименование/код → полное наименование**: пользователь просит «дай полное по СМ02/SM02»; нужно найти все строки прайса с этим кратким кодом и вернуть соответствующее полное наименование. Если строк несколько, покажи все варианты и, при необходимости, отличия.

Если пользователь поправил код товара, не спорь: перепроверь по `price_list_entries` и признай ошибку, если сопоставление было сделано нестрого.

## Обязательный workflow

1. Сначала прочитай скриншот через vision, выпиши:
   - полное наименование;
   - товарный знак/вариант исполнения;
   - все компоненты в скобках;
   - количество и цену, если видны;
   - НДС/единицу, если видны.
2. Сверяй только с прайсом в коннекторе:
   - `mcp_prostye_postavki_read_table_rows(table="price_list_entries", search=<ключ>, limit=100)`
   - ключи брать по товарному знаку и уникальным компонентам: например `ИХА-МУЛЬТИ-ПЛЮС`, `контролем подлинности образца`, `МДПВ ЭТГ`, `спайс`, `прегабалин`, `НТ-СКРИН`, `тест-полоска`.
3. Если найдено несколько похожих строк, сравнивай полный состав, а не один компонент:
   - количество видов;
   - наличие/отсутствие каждого компонента;
   - контроль подлинности: SG, CRE, pH;
   - цену и единицу измерения, если они помогают отличить позиции.
4. Не выдавай уверенный код, если состав не совпадает полностью. В таком случае скажи: «по прайсу точного совпадения не вижу, ближайшие варианты: …».
5. Для «ИХА-МУЛЬТИ-ПЛЮС» особенно не путать близкие позиции:
   - `СМ04`: 11 видов, в прайсе строка с морфин, марихуана, амфетамин, метамфетамин, барбитураты, бензодиазепин, кокаин, метадон, спайс, мдпв, ЭТГ + SG, CRE, pH.
   - `СМ09`: 11 видов, строка с фенциклидин вместо спайса: морфин, марихуана, амфетамин, метамфетамин, барбитураты, бензодиазепин, кокаин, метадон, фенциклидин, мдпв, ЭТГ + SG, CRE, pH.
   - `СМ12`: 13 видов, строка с фенциклидин, МДМА, спайс, мдпв, ЭТГ + SG, CRE, pH.
   Эти подсказки не заменяют сверку: каждый раз проверяй актуальный `price_list_entries`.
6. Финальный ответ держи коротким:
   - «Это <код>»;
   - 1 строка с причиной: «совпало по составу: …»;
   - если не уверен — явно пометь неуверенность и дай ближайшие варианты.

## Расчёт рабочих дней в договорах

Когда во входящем договоре срок задан в рабочих днях, считай рабочие дни по официальному производственному календарю РФ, а не просто по будням:

- исключай субботы и воскресенья;
- исключай официальные нерабочие праздничные дни и переносы выходных;
- сокращённый предпраздничный день остаётся рабочим, если он не объявлен выходным;
- перед сохранением срока сверяй конкретный год с производственным календарём, если в периоде есть праздники;
- пример: от 10.06.2026 срок 5 рабочих дней = 18.06.2026, потому что 12.06.2026 — День России, нерабочий праздничный день.

Если сервер автоматически считает рабочие дни без праздников и даёт неверную дату, сохраняй конкретный `deadlineDate` вручную и укажи в примечании, что срок рассчитан по производственному календарю РФ.

## Коммерческие предложения

Если Александр просит сформировать КП по короткому коду/артикулу из прайса и отправить его на email, действуй по `references/commercial-offers-by-price-code.md`: сначала прочитай MCP prompt `commercial_offer_workflow`, затем разреши код через актуальный `price_list_entries` (учитывая латиницу/кириллицу в `SM02`/`СМ02`), проверь количество/цену/НДС/шаблоны/получателя и только после этого вызывай инструмент создания/отправки КП.

Если запрос на КП пришёл в письме от Максима/клиента и содержит вложенную спецификацию с маркетплейс-ссылками, действуй по `references/commercial-offers-from-email-specs.md`: извлеки тело письма и вложения, отдели настоящего заказчика от переславшего, выпиши позиции/количества/дедлайн, рассчитай цены из спецификации/маркетплейсов по `commercial_offer_workflow`, а при создании архива не пытайся насильно сопоставлять обычные товары с медицинским прайсом.

## Входящие договоры и спецификации

Подробный конспект с примерами и ошибками см. в `references/incoming-contract-item-name-mapping.md`.

Когда обрабатываешь входящий договор/контракт со спецификацией:

- `docName` / «Наименование для документов» = полное наименование из спецификации договора, дословно в одну строку, с сохранением всех слов, скобок, марок и пробелов после нормализации переносов;
- `name` / «Наименование» = краткий код из загруженного прайса только при уверенном совпадении полного состава позиции с прайсом;
- если договорная позиция похожа на строку прайса, но состав/вариант/контроль подлинности не совпадает полностью — не подставляй код, оставь полное наименование и явно пометь неоднозначность;
- пример: позиция договора «Набор реагентов для диагностики ин витро ... десяти видов наркотиков ... (морфин, марихуана, амфетамин, бензодиазепин, барбитураты, кокаин, метамфетамин, метадон, фенциклидин, МДПВ, SG, CRE, показатель рН) ...» при совпадении со строкой прайса сохраняется как `name = СМ02`, а `docName` остаётся полным текстом из договора.

## Нюанс импорта прайса

- Одинаковое короткое название/код не означает дубль. В прайсе допустимы несколько строк с одним кодом, если полное наименование отличается.
- При проверке/правках загрузчика дубль нужно определять по нормализованной паре «код + полное наименование». Совпадает только код, а наименование другое → это отдельная строка, её нельзя перетирать.
- Если пользователь подозревает пропажу строк при загрузке прайса, используй диагностический сценарий из `references/price-import-duplicate-handling.md`: проверить живую таблицу, затем серверный upsert/поиск существующей записи, затем тестовый импорт двух строк с одинаковым кодом и разными полными наименованиями.
- Отдельно следи за кириллицей/латиницей в кодах (`СМ02` и `SM02` выглядят похоже, но это разные строки для поиска); в ответе называй фактический вариант из прайса.

## Для входящих договоров

При ручной обработке входящего договора/контракта со спецификацией:

- `docName` / «Наименование для документов» всегда оставляй полным наименованием из столбца «Наименование» спецификации — дословно, одной строкой, с сохранением пробелов, скобок, моделей, ТУ, страны происхождения и прочих частей строки.
- `name` / обычное «Наименование» НЕ копируй автоматически из договора, если в прайсе есть соответствие. Сначала ищи полное наименование из спецификации в загруженном прайсе/каталоге; при точном уверенном совпадении ставь краткое наименование/код из прайса, например `SM02`/`СМ02` — именно в том написании, как в прайсе.
- Если точного соответствия в прайсе нет или есть несколько разных вариантов по одному коду/похожему составу, не выбирай молча: оставь `name = docName` или переспроси/покажи варианты, в зависимости от сценария.
- Пример: если в спецификации длинная строка «Набор реагентов …» соответствует позиции прайса `SM02`, то сохраняй `name = "SM02"`, а `docName =` полная строка из договора.

## Запреты

- Не сопоставляй по прошлой памяти без вызова коннектора.
- Не выбирай код по одному слову, если есть близкие позиции.
- Не игнорируй цену/единицу, когда состав похожий.
- Не называй системные таблицы и технические детали пользователю без необходимости; просто скажи, что сверил с загруженным прайсом.

## Проверка перед ответом

Перед финалом убедись:

- код взят из загруженного прайс-листа;
- состав со скриншота совпал со строкой прайса или неуверенность явно указана;
- если пользователь уже поправил код, ответ учитывает его правку и объясняет, где была ошибка кратко
