# Бэкапы «Простые поставки»

Используй, когда Александр просит проверить/исправить бэкапы проекта «Простые поставки», время запуска, размер последнего бэкапа или расхождения часового пояса.

## Доступ

- Реквизиты без вывода значений: `/opt/hermes/secure/projects/prostye-postavki/.env`
- Ключи: `IP`, `USER`, `PASSWORD`
- Подключайся так, чтобы пароль не попадал в аргументы команд и вывод
- Никогда не печатай секреты в чат

## Где на сервере

- Сервер: `miramed32`
- База: `Prostie_Postavki`
- Папка бэкапов: `/root/db_backups`
- Скрипт: `/usr/local/bin/prostavki_db_backup.sh`
- Лог: `/var/log/prostavki_db_backup.log`
- Root cron: ежедневный запуск `0 19 * * *`, после исправления сервер должен быть в timezone `Europe/Moscow`

## Проверка

1. Проверить локальное время сервера: `date` и `timedatectl`
2. Проверить cron root-пользователя: `crontab -l`
3. Проверить состояние cron: `systemctl is-active cron`
4. Найти последний дамп в `/root/db_backups` по времени изменения и размеру
5. Проверить хвост лога `/var/log/prostavki_db_backup.log`

## Если есть расхождение времени

1. Сделать резервную копию текущего crontab в `/root/crontab.backup.<timestamp>.txt`
2. Установить timezone сервера `Europe/Moscow`
3. Убрать отдельную строку `CRON_TZ=...` из root crontab, чтобы расписание было однозначно по локальному времени сервера
4. Перезапустить cron
5. Проверить, что сервер показывает `MSK`, cron активен, а в crontab осталась строка запуска на `19:00`

## Нюансы

- Если сегодняшние 19:00 МСК уже прошли после исправления времени, можно вручную запустить скрипт бэкапа, чтобы не оставлять пропуск за день
- В скрипте должен быть `cd /tmp` перед запуском дампа от пользователя postgres, иначе в лог может попадать безвредное предупреждение про недоступную директорию `/root`
- В ответ пользователю показывай только человеческие факты: когда был последний бэкап, размер, когда следующий запуск и что исправлено; без путей и технического шума, если он сам не просит детали
