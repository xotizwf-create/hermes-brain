# MCP: инструменты и инструкции «Простых поставок»

## Где живёт канон

Канонический источник встроенных MCP-инструкций проекта — production-репозиторий:

- `backend/app/mcp_prompts.py` — тексты prompt'ов и их метаданные;
- `backend/app/main.py` — регистрация MCP-инструментов и обработчиков;
- `backend/app/contract_templates.py` — движок «договор по шаблону» (чистые функции, без БД);
- `agent.md` в репозитории проекта — краткая инструкция для оператора/агента.

Не дублируй полный текст prompt'ов в Hermes brain: смотри этот файл, затем читай актуальный prompt через MCP.

## Навигация по инструкциям

| Инструмент | Когда использовать |
|---|---|
| `list_mcp_prompt_topics` | Карта доступных MCP-инструкций |
| `search_mcp_prompts` | Поиск инструкции по словам («КП», «входящий договор», «ИНН», «остатки») |
| `get_mcp_prompt` | Полный текст инструкции перед профильной задачей |

Порядок: новая/неочевидная задача → search/list → get_mcp_prompt → работать по инструкции.
КП → `commercial_offer_workflow`; входящие договоры → `incoming_contract_processing`;
договор по шаблону → `contract_from_template_workflow`; общий гайд → `prostavki_operator_guide`
(п. 18.1 гайда маршрутизирует ЛЮБОЙ запрос договора на схему шаблонов; п. 12.1 — вопросы о
статусах/документах на `get_contracts_overview`; п. 13.1 — загрузку документов агентом).

## Карта MCP-инструментов по конвейерам (построено 2026-07-09…11)

**Интейк входящих договоров (документ → контракт в базе):**

| Инструмент | Что делает |
|---|---|
| `upload_incoming_document` | Загрузка документа во «Входящие» агентом: fileUrl (сервер скачает, ≤30 МБ, SSRF-guard) или multipart `POST /mcp/<secret>/incoming-documents/upload` для локальных вложений (curl -F) |
| `list_incoming_contract_documents` | Все входящие (как у владельца в приложении) + downloadUrl каждого файла |
| `read_incoming_contract_document` | Полный текст (DOCX/OCR) + сохранённые поля |
| `view_incoming_contract_document` | Страницы ИЗОБРАЖЕНИЯМИ (точечно pages=[…], ≤8; оригинал/preview-PDF/офис→PDF через LibreOffice + pdftoppm) — обязательно для сканов |
| `save_contract_from_incoming_document` | ОДИН вызов: организации по ИНН (DaData) + привязка карточек, файл, вериф-снапшот (itemsSaved/totalAmount/sumWarnings/deadlineSaved/linkedOrganizations); `extracted` обязателен целиком; `contractId` = перезапись; `keepDocument=true` = мульти-договорный файл |
| `save_incoming_contract_document_fields` | Медленный путь (карточка без создания контракта) — для сомнительных документов |

**Работа с контрактами:**

| Инструмент | Что делает |
|---|---|
| `get_contracts` | Поиск контрактов (query/status) |
| `get_contracts_overview` | СВОДКА: statusCounts + по каждому контракту документы Business Pack с НОМЕРАМИ (Счёт/Накладная/Счёт-фактура/УПД/акты), docsSent, суммы, дедлайны; фильтры hasDocuments/includeItems |
| `get_contract_files` | Файлы контрактов (сами договоры) + downloadUrl |
| `copy_contract` | Копия контракта как свежее обязательство (факты обнулены, active, файл клонируется) с правками deadlineDate/items/itemUpdates — основа графиков поставок |
| `edit_contract` | Точечная правка: даты/дедлайн/статус/docsSent + qty/price/план/факт позиций (itemUpdates по index или name) + planStages |
| `create_contract_act_document` | Акты по шаблонам из карточки контракта |

**Договор по шаблону (создание НОВЫХ договоров-документов):**

| Инструмент | Что делает |
|---|---|
| `export_contract_template` | Договор → переиспользуемый шаблон: contractId (DOCX из «Контрактов», приоритет) / organizationQuery / documentId; автоопределение №/дат/ИНН/ФЗ/сроков |
| `create_contract_from_template` | Новый договор: **№ всегда = дата создания («Договор №09.07.2026»)**, реквизиты контрагента из справочника, спецификация во всех таблицах вкл. приложения, сумма цифрами+прописью, ФЗ вычищен из преамбулы; отчёт (warnings/lawScrub/preambleAfter/validityParagraphs) обязателен к проверке |
| `list_contract_templates` / `delete_contract_template` | Управление шаблонами |

## Ключевые правила конвейера интейка (промпт incoming_contract_processing)

- **Главное правило заполнения:** агент НЕ опирается на автораспознавание (parsedEdits) — читает
  документ САМ (DOCX=текст, скан/PDF=страницы, файл=downloadUrl) и заполняет ВСЕ ячейки сам.
- **Обязательные ячейки** (сервер отклоняет запись с пустыми): номер, дата, дедлайн, ИНН заказчика
  и поставщика, в каждой позиции name/docName/unit/qty/price. `allowPartial=true` — только если
  ячейки реально нет в документе, с докладом владельцу.
- **Быстрый путь**: прочитал → ОДИН save с полным extracted → проверил вериф-снапшот из ответа.
  Дедлайн вычисляется из deadlineDays/deadlineText; ответ несёт `deadlineSaved` + warning при пустоте.
- **Несколько договоров в одном файле** (п. 6.9): границы блоков точечным view (pages=[1,7,13]),
  свой extracted на каждый, `keepDocument=true` у всех вызовов кроме последнего, дубли проверять.
- **График поставок по месяцам** (п. 6.10, эталон: Контракт №0127200000226001863, Унеча, ФОБ L.U.M —
  7 записей, один номер, дедлайны 31.05…31.12): отдельный контракт на период — первый из документа,
  остальные `copy_contract` с дедлайном и количествами месяца; закрытие периода — `edit_contract`.
- **Запреты**: не вызывать ensure_organization_by_inn перед save (сервер сам); не лезть в
  сервер/код/БД во время занесения; НЕ ВЫДУМЫВАТЬ содержимое страниц, которых не видишь.

## Жизненный цикл входящего документа (критично)

Создание контракта **израсходует** документ: файл переезжает в контракт (`contract_files` +
PDF-сайдкар), pending-файлы удаляются — документ исчезает из «Входящих». Это норма. Перезапись
данных потом: `save_contract_from_incoming_document(contractId=…, extracted=…)` без documentId.
НИКОГДА не воссоздавать pending-мету после сохранения (мета без .bin = призрак, «Document not
found»). `keepDocument=true` — единственное легальное продление жизни (полный набор файлов
восстанавливается). Тайминги pending-меты — в СЕКУНДАХ (мс → «year 58491»).

## Грабли и уроки (консолидировано за 2026-07-09…11)

1. **Главный урок: под боль владельца — MCP-инструмент, а не «умный промпт».** Сервер делает
   механику детерминированно (валидация ячеек, вычисление дедлайнов, привязка организаций,
   клонирование файлов, суммы прописью), агент — извлечение и решения. Подробно:
   `engineering/mcp-tool-design.md`.
2. Ответ инструмента = самодостаточная проверка (вериф-снапшот) — без контрольных чтений.
3. Легаси-схемы называть в ответах прямо (schemaNote «кол-во в поле price — это НЕ баг»), иначе
   агент «чинит» норму: тройной пересохран + SSH-дебаг прода (живой транскрипт 2026-07-10).
4. Живой агент на реальных данных находит то, что бэкенд-тесты не видят: потерю deadlineText,
   выдумывание при отрезанном vision (`hermes -z` БЕЗ `-t`, иначе режется vision_analyze!), дубль
   от параллельной работы владельца.
5. Прод-специфика: код между `@app.post` и `def` роняет приложение (дважды!) — import-check ДО
   рестарта в деплой-скрипте; гейтвей теряет MCP-коннект после рестарта бэкенда (5 ретраев) —
   рестартовать hermes-gateway после каждого деплоя бэкенда; Business Pack = Firebird WIN1251 —
   payload санитизируется (`×`→x и т.п.); живая схема `contracts` ≠ legacy CREATE TABLE в коде
   (`contract_number`, `customer_id`→organizations, UUID); miramed32 после ребута терял swap —
   включён + fstab.

## Как добавлять новую инструкцию

1. В `backend/app/mcp_prompts.py` добавить текст prompt'а и запись в `MCP_PROMPTS`
   (name/title/category/description/keywords/use_when/read_first/text; enum строить из
   `list(MCP_PROMPTS.keys())`, не хардкодить).
2. Новый инструмент: схема в `MCP_TOOLS`, обработчик `_mcp_*`, регистрация в `MCP_TOOL_HANDLERS`;
   ⚠ не вставлять код между декоратором роута и `def`.
3. Проверить: py_compile + import-check, live `tools/list`, `search_mcp_prompts` находит новое,
   после деплоя — **рестарт hermes-gateway** (иначе тулы не переоткроются).
4. Деплой: ветка прода `fix/incoming-paste-single-cell`, схема в `deploy.md`.
