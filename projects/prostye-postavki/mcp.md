# MCP-инструкции и навигация

## Где живёт канон

Канонический источник встроенных MCP-инструкций проекта «Простые поставки» находится в production-репозитории:

- `backend/app/mcp_prompts.py` — тексты prompt’ов и их метаданные;
- `backend/app/main.py` — регистрация MCP-инструментов и обработчиков;
- `agent.md` в репозитории проекта — краткая инструкция для оператора/агента.

Не дублируй полный текст prompt’ов в Hermes brain: сначала смотри этот файл, затем читай актуальный prompt через MCP.

## Инструменты навигации

Для быстрого выбора инструкции используй read-only MCP-инструменты:

| Инструмент | Когда использовать |
|---|---|
| `list_mcp_prompt_topics` | Показать карту доступных MCP-инструкций: название, категория, ключевые слова, когда читать |
| `search_mcp_prompts` | Найти нужную инструкцию по словам вроде «КП», «Интеко», «входящий договор», «ИНН», «остатки» |
| `get_mcp_prompt` | Прочитать полный текст выбранной инструкции перед выполнением профильной задачи |

Общий порядок:

1. Если задача новая или неочевидная — вызвать `list_mcp_prompt_topics` или `search_mcp_prompts`.
2. Перед выполнением профильной операции прочитать полный prompt через `get_mcp_prompt`.
3. Для коммерческих предложений обязательно читать `commercial_offer_workflow`.
4. Для входящих договоров/контрактов обязательно читать `incoming_contract_processing`.
5. Для общих вопросов по остаткам, контрактам, поставкам, организациям и инструментам начинать с `prostavki_operator_guide`.

## Договор по шаблону (добавлено 2026-07-09, источник «Контракты» — 2026-07-10)

Быстрое создание нового договора из существующего договора-DOCX. Движок:
`backend/app/contract_templates.py` (чистые функции, без БД) + MCP-обвязка в `main.py`.
**Главный источник исходных договоров — раздел «Контракты»** (файлы в `contract_files` +
`contract_files_store/`): 33 реальных DOCX на 2026-07-10. Входящие документы — запасной источник.
⚠ Живая схема `contracts` ≠ legacy CREATE TABLE в коде: реально `contract_number`,
`customer_id`/`supplier_id` → organizations, `comment`, UUID id.

| Инструмент | Что делает |
|---|---|
| `get_contract_files` | Файлы, прикреплённые к контрактам (сами договоры DOCX/PDF): по contractId или query (номер/заказчик/имя файла); отдаёт isDocx и downloadUrl (`/mcp/<secret>/contract-files/<id>` — те же файлы, что кнопка скачивания в приложении) |
| `export_contract_template` | Выгрузить договор как шаблон: contractId (DOCX из «Контрактов», приоритет) / organizationQuery (сначала контракты, потом входящие) / documentId; сервер сам определяет №, дату, ИНН сторон, упоминания ФЗ и сроков. PDF/сканы шаблоном не становятся |
| `create_contract_from_template` | Новый договор из шаблона: **№ всегда = дата создания («Договор №09.07.2026»)**, реквизиты контрагента из справочника организаций, спецификация перестраивается во всех таблицах (включая приложения), сумма пересчитывается цифрами+прописью, ФЗ вычищается из преамбулы; возвращает отчёт (warnings/lawScrub/preambleAfter/validityParagraphs) — агент обязан его проверить и добить промахи через `replacements` |
| `list_contract_templates` / `delete_contract_template` | Список/мягкое удаление шаблонов (таблица `contract_templates`) |

Бизнес-правила зафиксированы в prompt `contract_from_template_workflow` (читать перед работой);
операторский гайд (п.18.1) маршрутизирует ЛЮБОЙ запрос договора на эту схему.
Готовый файл отдаётся по `downloadUrl` (`/mcp/<secret>/contract-exports/…`). Смоук-тест движка:
`backend/tools/contract_template_smoke.py` (гоняется и локально, и на проде при деплое); в нём же
реальный канон преамбулы (223-ФЗ, латинская N, «2026г.», двойные пробелы). Замены дат идут с
`flexibleWhitespace` (движок терпит разрывы пробелов), варианты номера — optional-группой без шума.

## Быстрый интейк входящих договоров (2026-07-10)

Конвейер «входящий документ → контракт в базе» ускорен и уточнён:

- **Один вызов вместо цикла**: `save_contract_from_incoming_document(documentId, extracted={…})` за
  раз: заводит недостающие организации по ИНН (DaData) и **привязывает их КАРТОЧКИ к контракту**
  (`customer_id`/`supplier_id` → реальные организации, а не системная заглушка — это чинилось в
  `create_contract`/`update_contract` через `_resolve_contract_party_org_id`), прикрепляет файл,
  помечает документ обработанным (`processed_at`, из дефолтного списка уходит) и возвращает
  вериф-снапшот: `contract` + `linkedOrganizations` (с флагами `isPlaceholder`) — контрольные чтения
  не нужны, если снапшот сошёлся.
- **Документ «как есть»**: `view_incoming_contract_document(documentId, firstPage, maxPages≤8)` —
  страницы ИЗОБРАЖЕНИЯМИ (JPEG ~130dpi): оригинал-картинка / сохранённый preview-PDF / офисный файл
  конвертится на лету (LibreOffice) → растеризация `pdftoppm` (poppler-utils, установлен 2026-07-10).
  MCP-диспетчер поддерживает multimodal-ответ (`_mcp_raw_content` → image content blocks).
  Обязателен для сканов/PDF и сомнительного OCR.
- Промпт `incoming_contract_processing`: пп. 6.1–6.4 — быстрый путь + правило «смотри глазами»;
  медленный путь (save fields → контрольные чтения) остался для сомнительных документов.
- ⚠ Тайминги pending-меты — в СЕКУНДАХ (uploaded_at/processed_at); мс ломают
  `datetime.fromtimestamp` («year 58491»).
- **Анти-луп интейка (2026-07-10, по живому транскрипту Hermes):** агент терял 10+ минут на
  договоре из-за трёх ловушек — (1) легаси-схема parsedEdits (кол-во в поле `price`, цена в
  `total`) выглядела как «искажение данных» → тройное пересохранение → SSH-дебаг прода;
  (2) каждая страница view = отдельный vision_analyze (8 страниц = 8 вызовов);
  (3) лишние ensure_organization_by_inn перед save. Фиксы: ответы save-инструментов несут
  `itemsReadable`/`itemsSaved`/`totalAmount`/`sumWarnings` (нормальные qty/price + серверная
  сверка сумм) и `schemaNote`; view принимает `pages=[…]` (точечные страницы, дефолт 3);
  промпт пп. 6.5–6.8: без pre-save ensure, схема-не-баг, **жёсткий запрет SSH/код/БД во время
  занесения**, перезапись контракта = один вызов с contractId. Фантомные меты без .bin
  отфильтрованы из list_incoming_contract_documents.
- ⚠ Урок 2026-07-10: вставка кода между `@app.post(...)` и `def create_contract` уронила прод
  (декоратор навесился на хелпер) — откат за ~2 мин; теперь деплой-скрипт делает **import-check до
  рестарта** (падает — сервис остаётся на старом коде).

## Жизненный цикл входящего документа (выяснено 2026-07-10 — критично для агента)

Создание контракта **израсходует** входящий документ: `_attach_pending_file_to_contract` переносит
файл в стор контракта (`contract_files` + PDF-превью сайдкаром) и УДАЛЯЕТ pending-файлы — документ
исчезает из «Входящих» и в приложении, и в MCP. Это норма, не потеря. Следствия:
- Перезапись уже внесённых данных: `save_contract_from_incoming_document(contractId=…, extracted=…)`
  **без documentId** (обработчик строит минимальную мету сам; файл контракта сохраняется).
- НИКОГДА не воссоздавать pending-мету после сохранения (мой первый вариант «пометить processed»
  порождал мету-призрак без .bin — «Document not found», агент слеп). Обработчик помечает processed
  только если pending-файлы почему-то уцелели.
- «Агент не видит входящие» имеет две типовые причины: (1) документ израсходован — он в контракте,
  смотреть get_contract_files; (2) гейтвей потерял MCP-коннект после рестарта бэкенда (5 ретраев ≈30с
  и отключение навсегда) — нужен рестарт `hermes-gateway`.
- Сервер miramed32 перезагружался 2026-07-10 и swap не поднялся сам — включён и добавлен в fstab.

## Интейк v3 (2026-07-10, проверено ЖИВЫМ агентом на реальном PDF)

- **Скачивание входящих**: каждый документ в list/read несёт `downloadUrl`
  (`/mcp/<secret>/incoming-documents/<id>` — файл как есть). Единый принцип для всех форматов:
  DOCX = read (текст), PDF/скан = view (страницы-изображения), сам файл = downloadUrl.
- **Обязательные ячейки** (сервер отклоняет запись с пустыми): номер, дата, дедлайн, ИНН заказчика
  и поставщика, в каждой позиции name/docName/unit/qty/price. `allowPartial=true` — только если
  ячейки реально нет в документе, с явным докладом владельцу.
- **extracted обязателен**: parsedEdits/автораспознавание больше НЕ источник — агент заполняет всё
  сам из документа (правило владельца). Быстрый путь вычисляет дедлайн из deadlineDays/deadlineText
  (раньше молча терял → пустая ячейка); ответ несёт `deadlineSaved` + warning при пустоте.
- **Business Pack**: payload глубоко санитизируется в cp1251 (`_bp_sanitize_deep`): × ✕ — … Ø µ ½ →
  безопасные замены (Firebird WIN1251 падал на «10×10»).
- **Живой тест агента** (hermes -z, только toolset prostye_postavki): реальный PDF «9538 садовый
  инвентарь» обработан за один ход — номер/стороны/3 позиции/24899₽ сошлись с документом (сверял
  страницы глазами через view). Найденное по ходу: дубль (владелец создал карточку через UI
  параллельно) удалён, карточка владельца перезаписана одним вызовом с contractId
  (deadline 31.07.2026 вычислен из «21 календарный день»).
- ⚠ Дважды наступили: вставка кода между `@app.post` и `def` — ВСЕГДА проверяй, что над функцией
  нет декоратора; import-гард в деплое ловит это до рестарта.

## Несколько договоров в одном файле (2026-07-11)

`save_contract_from_incoming_document(..., keepDocument=true)` — документ переживает сохранение
(pending-файлы снапшотятся до attach и восстанавливаются после, БЕЗ пометки processed) и остаётся
во входящих для следующего договора; файл прикрепляется к КАЖДОМУ контракту; последний вызов — без
keepDocument (документ уходит). Промпт п. 6.9: границы блоков ищутся точечным view (например
pages=[1,7,13] для 18 страниц), у каждого договора СВОЙ extracted, перед созданием — проверка
дублей через get_contracts. Механика проверена e2e (MULTI_CONTRACT_OK), инструкция — живым агентом
на реальном 18-страничном скане с 3 контрактами (границы/номера/суммы/сроки совпали с ручной
сверкой страниц).

⚠ **Антивыдумывание (урок 2026-07-11):** в тестовой -z сессии с `-t prostye_postavki` у агента НЕТ
встроенного vision_analyze → изображения view не доходят до модели, и агент УВЕРЕННО ОПИСАЛ
содержимое скана из своей памяти (прошлые контракты). В проде (Telegram-ход, полные тулсеты)
vision работает и чтение честное. Фиксы: жёсткое правило в hint view-инструмента и промпте 6.1 —
«описывать можно только то, что реально видишь; изображения не дошли → остановись и скажи
владельцу». Агентские тесты через `hermes -z` гонять БЕЗ `-t`-ограничения (иначе режется vision).

## График поставок по периодам + copy/edit контрактов (2026-07-11)

Практика владельца (эталон: Контракт №0127200000226001863, Унеча, ФОБ L.U.M — семь записей, один
номер, дедлайны 31.05…31.12, количества своего месяца, прошлые completed / будущие active):
договор с графиком поставок = ОТДЕЛЬНЫЙ контракт на каждый период. Промпт п. 6.10.

| Инструмент | Что делает |
|---|---|
| `copy_contract` | Новый контракт копией существующего: факты обнулены, статус active, файл договора клонируется; правки — deadlineDate, items (замена) или itemUpdates ({index\|name, qty, price…}), number/contractDate/status/planStages |
| `edit_contract` | Точечная правка без полного extracted: номер/даты/дедлайн/статус/docsSent, itemUpdates (qty/price/план/факт по index или name), planStages |

Поток: первый период — save_contract_from_incoming_document (+keepDocument при необходимости),
остальные — copy_contract с дедлайном и количествами периода; закрытие периода — edit_contract
(status=completed, factQty/factDate). E2e: SCHEDULE_TOOLS_OK (3 записи одного номера, клон файла,
edit цены/статуса/фактов, полная очистка).

## Как добавлять новую инструкцию

1. В `backend/app/mcp_prompts.py` добавить текст prompt’а и запись в `MCP_PROMPTS`.
2. В записи заполнить:
   - `name` — стабильный машинный id;
   - `title` — короткое человеческое название;
   - `category` — группа сценариев;
   - `description` — что это за инструкция;
   - `keywords` — слова для поиска;
   - `use_when` — когда читать;
   - `read_first` — `true` только для стартового общего гайда;
   - `text` — полный текст prompt’а.
3. Не хардкодить список prompt’ов в schema `get_mcp_prompt`: enum должен строиться из `list(MCP_PROMPTS.keys())`.
4. Если появляется новый сценарий, добавить ключевые слова так, чтобы `search_mcp_prompts` находил его по обычному языку владельца.
5. Проверить:
   - синтаксис Python;
   - `list_mcp_prompt_topics()` возвращает новый prompt;
   - `search_mcp_prompts(<живой запрос>)` ставит нужный prompt в верх выдачи;
   - live MCP `tools/list` видит новые/изменённые инструменты после перезапуска backend.
