# Albery — план разреза монолитов (утверждён владельцем 2026-07-02)

## Контекст
Аудит 2026-05-26 (`audit-report.md` в репо Albery) закрыл 22 из 23 пунктов (PostgreSQL-only,
пул соединений, миграции как единственный источник схемы, security, CI). Остался пункт 1.4 —
god-объекты, и это цель данного плана:

| Монолит | Размер (на 2026-07-02) | Внутри |
|---|---|---|
| `app.py` | 23 519 строк, 110 маршрутов, ~740 функций | LLM, синки Bitrix/Zoom/Drive, отчёты, auth, MCP-over-HTTP, Bitrix-чатбот |
| `mcp/context_server.py` | 5 900 строк | ~50 MCP-инструментов |
| `Интерфейс/src/App.tsx` | 9 227 строк, один компонент | ~125 useState, ~40 «голых» fetch |

Монолит растёт (23 284 → 23 519 с момента аудита): новый код (b24-бот, уведомления) ложится туда же.
Тестов 15 файлов — до разреза нарастить сетку. Прецедент риска: чистка 2026-05-26 вырезала
19 живых функций и уронила оба процесса (см. журнал в `audit-report.md`) — поэтому только
перемещение кода, никакого переписывания, каждый шаг проверяем.

## Шаги (каждый — отдельный PR)
- **Ш0. Предохранители:** прод на 186 == HEAD (`git status` на сервере), бэкап БД, деплой
  атомарный с откатом, `engineering/server-preflight.md` обязателен. Никаких тестов на живой БД.
- **Ш1. Страховочная сетка:** характеризационные тесты на чистые функции (парсинг Zoom-задач,
  нормализации, форматтеры), смоук «`import app` + маршруты отвечают», pytest в GitHub Actions.
- **Ш2. Разрез `app.py`** (только move, от периферии к ядру):
  1. `b24bot/` — Bitrix-чатбот (~строки 21 655 — конец, самый обособленный кусок);
  2. `config.py` — env и константы;
  3. `integrations/` — bitrix, zoom, google_drive;
  4. `llm/` — провайдеры и промпты;
  5. `services/` — синки и генерация отчётов (owner / chats / zoom);
  6. `routes/` — Flask-блюпринты (auth, api, mcp-http) — последними.
  Проверка каждого шага: `py_compile` + `pyflakes` (0 undefined) + тесты + смоук на сервере.
- **Ш3. Разрез `mcp/context_server.py`:** инструменты по доменам (`mcp/tools/…`) поверх общих
  репозиториев в `shared/` — добивает пункт 1.3 аудита (дубли логики app ↔ MCP).
- **Ш4. Фронтенд:** `App.tsx` → экраны (Registry / Reports / Zoom / Chats / Owner / Настройки
  агента); все fetch через `api/client.ts` (`fetchJsonSafe`), долгим LLM-POST — явный `timeoutMs`
  (иначе таймаут оборвёт легитимную генерацию — известный риск из аудита). Проверка:
  `tsc --noEmit` + `vite build` + ручной прогон.
- **Ш5. Правило:** `app.py` заморожен на приём нового кода — новое только в свои модули.

- **2026-07-02 Ш2.4 СДЕЛАНО:** Bitrix-интеграция вынесена в `bitrix.py` (коммит `5e25710`,
  33 имени / ~1 950 строк: BitrixClient, вебхуки task/team-событий, очередь событий,

  (`pg_connect`/`pg_json`/`pg_table_exists`); остальное — из config/llm/utils. app.py внизу
  делает `import bitrix` + `from bitrix import <18 имён>` (re-import в свой неймспейс — обратные

  проверка для этого паттерна). Тесты: два bitrix-sync теста переведены на новую фикстуру
  `bitrix_module` — **патчить надо модуль, где код живёт** (bitrix держит свои привязки
  BitrixClient/pg_connect; патч на app их не меняет — эта грабля теперь задокументирована в
  conftest). 80 тестов, CI зелёный, прод: рестарт, вебхуки отвечают, журнал чист, реальные
  Bitrix-события шли с кодом 200.
