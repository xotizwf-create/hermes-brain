## 2026-07-13 — два хода сотрудников зависли на все 600 сек (Наталья #669, Александр #709)

- **Impact:** Наталья (описание регулярной задачи со ссылками на Google-таблицы, 11:39) и Александр («Пришли ссылку на папку о компании», 13:22) прождали по 10 минут и получили отказ с литеральными `\n` в тексте и ложным обвинением Google-таблиц. За 14 дней таких ходов было 12.
- **Root cause:** зависший stream к ChatGPT/Codex-бэкенду. В state.db у обеих сессий ровно ОДНО сообщение (вопрос) — модель не выдала ни одного события за 10 минут; ни одного MCP-вызова в журнале. Параллельные ходы в те же минуты работали → сбой одиночного запроса, а не бэкенда целиком. Дефолтный per-call timeout hermes = 1800с; stale-детектор (90с) покрывает только «тишину до первого байта» (известная болячка codex silent-hang, hermes-agent #21444) — стрим, замолчавший ПОСЛЕ старта, висел бесконечно, и b24-кап 600с срабатывал первым. Ретрая на таймаут в b24bot нет (только на быстрый фейл).
- **Fix:** `/root/.hermes/config.yaml` → `providers.openai-codex.request_timeout_seconds: 240` (бэкап `config.yaml.bak-stall-timeout-20260713`). Для стримового Codex Responses это лимит ПАУЗЫ между событиями стрима: мёртвый стрим умирает за 240с → SDK кидает ReadTimeout → hermes ретраит вызов ВНУТРИ хода (`api_max_retries: 2`), не теряя уже сделанные tool-вызовы; живые длинные генерации (договоры юриста по 400-600с) шлют события и не задеты. Проверено: `get_provider_request_timeout('openai-codex', 'gpt-5.6-terra') = 240.0`, живой ход отвечает. CLI-ходы (все битрикс-агенты) читают конфиг на каждый запуск; hermes-gateway перезапущен для обновления кэша.
- **Заодно:** тексты таймаут-сообщений b24bot (Albery `4b818b4`) — реальные переносы строк вместо литеральных `\n`, честное «чаще всего это разовый сбой — отправьте ещё раз» вместо обвинения таблиц.
- **Осталось как защита:** жёсткий кап 600с в b24bot + ops-алерт «таймаут хода» в TG.
- **Как диагностировать такое:** state.db → sessions по времени старта (`id = YYYYMMDD_HHMMSS_*` МСК) → messages: одна user-запись без продолжения = зависший первый LLM-вызов; «hermes timed out after 600s» в журнале albery; mcp_tool_call-записей нет.

## 2026-07-12 — «Полина»: агент назвал ответственным человека, которого нет в компании

- **Impact:** в рекомендациях по новостной сводке агент указал «Полину» ответственной за P&L. Такой сотрудницы в активной команде нет — владелец заметил и потребовал разбор.
- **Root cause (двойной):** (1) в базе жили данные ДВУХ порталов — прежнего `damaska.bitrix24.ru` и текущего `b24-0xrp3s`; среди пользователей висели 28 карточек damaska (все неактивные), включая Полину Жуковскую (31147), плюс устаревшие Drive-документы «Карта процессов модели ИУ 04.05.26» с формулировкой «ДДС / баланс / P&L — Артур, Полина аутсорсный финменеджер». (2) Главное: агент взял Полину **из своего же ответа от 10.07** и перенёс дальше как установленный факт — история диалога инжектится в промпт, и модель приняла собственные слова за данные.
- **Fix:** вычищены все данные старого портала (чат-переписки целиком + 28 юзеров damaska; бэкап `albery_20260712_165539.dump`, одна транзакция, преф-проверки = 0 живых связей), ресинк с текущего портала (24 чата / 21 сообщение / 19 юзеров, 17 активных). Задачи НЕ тронуты — все 342 с нового портала (ложный признак: слово damaska в описании = историческая ссылка при переносе; ответственные в них — id нового портала).
- **Prevention:** universal-инструкция «Базовое поведение / Люди и роли — только актуальная оргструктура» (albery `2ae0c81`): свой прошлый ответ — не источник; ответственный только активный в `get_org_structure`; документ ≠ текущая ответственность; ссылки на damaska в задачах — история. Проверено боем: «кто такая Полина» → «в актуальной оргструктуре такой нет».
- **Признаки порталов** (для будущих чисток): старый — `bitrix_user_id >= 100` (5-значные) и `bitrix_chat_id >= 10000`; текущий — id пользователей 1..62.
- **Открыто:** устаревшие Drive-документы правятся только в самом Google Drive (у агента read-only токен).

## 2026-07-02 → 2026-07-12 — батч-синк молча умирал 10 дней (чаты и снапшоты задач устарели)

- **Impact:** реестр чатов стоял с 30.06, снапшоты задач/period-export и drive-синки — с 02.07.
  Живые задачи НЕ пострадали: bitrix_tasks обновляются событиями Bitrix inline
  (`BITRIX_TASK_EVENT_PROCESS_INLINE=1`) + кнопка «Обновить из Bitrix» — поэтому в UI всё
  выглядело свежим и поломку никто не заметил. Zoom тоже жил своим событийным путём.
- **Root cause:** рефакторинг ~02.07 вынес `sync_google_drive_company_documents` и
  `sync_google_drive_call_transcripts` из `app.py` в `gdrive.py`; hourly-cron
  `/etc/cron.d/albery-daily-sync` → `scripts/run_daily_sync.py` продолжал звать `app.sync_…`
  → `AttributeError` при построении списка шагов, т.е. ДО запуска любого шага — падали
  и чаты, и снапшоты, и team-синк. Traceback уходил только в
  `/var/log/albery/daily-sync.cron.log`; старый selfcheck смотрел лишь на ходы бота.
  Ночью 12.07 гейтвей-агент собрал отдельный systemd-таймер для drive-доков (работает),
  но остальной батч не чинил.
- **Fix (12.07):** `run_daily_sync.py` импортирует `gdrive` (albery-репо `4844321`);
  догоняющий прогон `AUTO_SYNC_CHAT_LOOKBACK_DAYS=13` — все шаги success, данные догнаны
  (чаты/снапшоты/sync_runs = 12.07 14:25 МСК).
- **Prevention:** `albery_selfcheck.py` расширен и проверен боем: свежесть батча по его же
  `daily-sync.log` (run_finished ≤2ч и success=true), systemd failed units, диск ≥85%,
  падения hermes-cron джоб (дедуп по сигнатуре), антиспам «тот же набор проблем — не чаще
  раза в 6ч» (state `/var/log/albery/selfcheck_state.json`). До фикса алерт реально ушёл
  в TG (батч + owner-weekly), после фикса — clean. Бэкапы: `/root/backups/*.bak-20260712`.
  Урок: при выносе функций из app.py грепать `app.<имя>` по scripts/ и кронам.
