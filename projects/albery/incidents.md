## 2026-07-14 — агент залипал на недоступных артикулах WB, ход умирал (600с)

- **Impact:** владелец: «Пройдись в таблице по всем артикулам, заполни цены» → ход умер по таймауту 600с (#760); «следующие 10» → 404с (~40с/артикул). Требование: агент не должен тупить на одном элементе, если остальные тянутся; недоступное — пропускать и честно называть.
- **Root cause (3 слоя):** (1) недоступный артикул ел 60-90с — до 6 поисковых комбинаций × пейсинг × 429-бэкоффы, без лимита времени на элемент; (2) модель тянула каждый артикул отдельным вызовом fetch_url — 43 круга модель↔инструмент; (3) весь поиск WB ходил с одного VPN-IP, чья 429-квота сгорала на списках → «цены не находились» у всех подряд (собственные тесты сжигали её же).
- **Fix (Albery `dd7a619`, Bitrix 1426):**
  1. **Метрика в коде:** дедлайн на элемент (12с в пакете / 25с одиночный) сквозь все шаги; негативный кэш — подтверждённый промах 10 мин, «медленный» элемент 3 мин (повтор = 0.3с).
  2. **`get_wb_prices`** — батч-инструмент (до 15 артикулов/вызов, бюджет вызова 45с, остаток → `skipped_by_budget`); в core + main/юрист/финансист + FAQ.
  3. **Поиск WB: прямой RU-IP первым** (отдельная 429-квота), VPN — фолбэк; устойчивый 429 → глобальный кулдаун 120с + честный статус `wb_throttled` («повторите через 3-5 минут»).
  4. **Маршрутная карта (hot, все агенты):** «НЕ ЗАЛИПАЙ НА ОДНОМ ЭЛЕМЕНТЕ» — для любых списков: макс. одна повторная попытка → пропусти → в конце честно перечисли с причиной; частичный результат ценнее нулевого; массовые цены WB → get_wb_prices пачками.
- **Проверено:** батч 5 (2 мёртвых) = 29с; e2e живым ходом список из 8 (2 мёртвых) = **55с**, цены собраны, «Недоступны: … WB сейчас не отдаёт цену» (раньше — смерть хода на 600с).
- **Урок:** «пропусти и иди дальше» нельзя оставлять на совести модели — бюджет времени, кэш промахов и батч-API должны быть в инструменте; промпт лишь договаривает честную подачу.

## 2026-07-14 — таблица цен WB заполнена историей цен вместо витринных (Софья)

- **Impact:** Софья: «ИИ Агент неправильно читает информацию с ВБ, таблицу заполняет неправильно». Таблица «Цены Wildberries» (43 артикула, «цена от/макс») содержала цены с копейками («1129,62»), которых на витрине не бывает, и неверные диапазоны; структура съехала (ссылка задублирована в A и B, мусор от повторных заполнений). Пример: 970473321 — в таблице «1129,62–1280,42», на витрине 1256 ₽.
- **Root cause:** после закрытия страниц WB антиботом (13.07) сводка карточки имела из ценового только ИСТОРИЮ цен CDN; модель выдавала мин/макс истории (изменения ПО ДАТАМ) за «цену от/до» (разброс ПО РАЗМЕРАМ). Формулировка сводки «История цен: мин…макс…» сама провоцировала подмену.
- **Fix (Albery `4c870c1`):** настоящий источник текущей цены — `search.wb.ru` (НЕ под антиботом): запрос бренд+название → название → артикул продавца, матч nm в выдаче, цены по размерам из `sizes[].price.product` + живой рейтинг. Грабли, вшитые в код: (1) поиск режет частые запросы **HTTP 429** → троттлинг `_wb_search_paced` (интервал `WB_SEARCH_MIN_INTERVAL_S`=2с, бэкофф-ретраи) — без него bulk-прогоны списков артикулов молча теряли цены; (2) карточка без остатков в регионе по умолчанию **скрыта из его выдачи** → фолбэк по регионам `WB_SEARCH_DESTS` (кейс «горошка» 791046510: Москва не отдаёт, другой регион → 542–799 ₽). История цен в сводке теперь подписана «изменения ПО ДАТАМ — НЕ разброс по размерам», при отсутствии живой цены — прямой запрет выдавать историю за диапазон.
- **Таблица Софьи перезаполнена** корректно: 39/43 с витринными ценами (целые рубли), 4 честно пустые (карточки скрыты из выдачи), структура починена. Bitrix-задача 1410.
- **Урок 1:** обходной источник данных обязан ЯВНО маркировать, чем он НЕ является — иначе модель подставит его в ближайший по смыслу слот (история цен ≠ диапазон размеров).
- **Урок 2 (офлайн-скрипты):** импорт `mcp.context_server` тянет `agent_center` и СТАРТУЕТ живые планировщики (12:00 чекин запустился из моего фонового скрипта; атомарный клейм по дате спас от дубля). Офлайн-скрипты на проде обязаны ставить `B24_TASK_CHECKIN=0` и `B24_TASK_OFFER=0` до импорта.

- **Impact:** Анастасия Андрусяк прислала xlsx фондовой политики (бот прочитал и ответил), следом отдельным сообщением задала вопрос «Во вложении фондовая политика… можно ли…» — бот ответил «вложение не отобразилось, пришлите заново». Владелец: «почему договоры не читаются?!»

Скоуп по agent_slug (dialog_id общий между ботами!). Блок не добавляется, когда к текущему сообщению приложены файлы.
- **Проверено:** смоук на её реальном диалоге (4 файла находятся, скоуп-изоляция, блок не дублируется), e2e-реплей провального хода через живой мозг → «использую файл att_MDS7TsNHfVF9» вместо «пришлите заново». Bitrix-задача 1404.
- **Урок:** если пользовательский паттерн (файл→отдельный вопрос) ломается на «внимательности модели» — чинить инжектом данных в промпт кодом, а не усилением формулировок.

## 2026-07-13 — бот дублировал ответы в личном чате (двойные сообщения у Александра)

- **Impact:** в чате с Александром Никитенко бот отвечал ДВАЖДЫ на одно сообщение («Сможем в нем сделать изменение?» → два одинаковых «Да, сможем…», строки #724/#725), плюс дважды показал апологию «я перезапустился». Владелец прислал скриншот, срочный фикс.
- **Root cause:** обработчик личных сообщений `ONIMBOTMESSAGEADD` (app-путь `_bitrix_imbot_app_event`) НЕ имел дедупа по `MESSAGE_ID` — в отличие от комментариев к задачам (`_b24_task_comment_claim`). Bitrix доставляет imbot-события **at-least-once** (повтор при сетевом сбое/ретрае), и каждая доставка запускала фоновую обработку → дубль ответа. Вторично: recovery после рестарта (`bitrix_inflight_turns`) слал апологию по КАЖДОЙ строке прерванного хода, а не по диалогу (я сегодня рестартовал сервис несколько раз при деплоях).
- **Fix (Albery `4c8e309`):** атомарный claim входящего по `MESSAGE_ID` (таблица `bitrix_bot_message_seen`, миграция 051, `INSERT … ON CONFLICT DO NOTHING RETURNING` ДО любого сайд-эффекта) — повтор молча игнорируется; fail-open (сбой БД → сообщение не теряется). Recovery-апология теперь ≤1 на диалог (`apologized`-сет по (bot,dialog)). Смоук: первая доставка=обработать, дубль=пропустить, другой id=обработать, пустой id=fail-open. Рестарт при inflight=0, журнал чист.
- **Урок:** ЛЮБОЙ вход от Bitrix (imbot-сообщения, task-комментарии, события) обрабатывать идемпотентно — claim по стабильному id перед сайд-эффектом. Комментарии это уже имели, личные сообщения — нет.
- Бэкап: `/root/albery-backups-20260713/b24bot.py.pre-patch6`.

## 2026-07-13 — агент завалил Оксану незваными ответами в задаче (оффер-фолбэк)

- **Impact:** Оксана Хапова (id 32) в задаче 1356 писала обычные рабочие комментарии (номера заявок Ozon) и закрыла задачу — агент ответил на КАЖДЫЙ комментарий (6 раз за 3 минуты), продолжая требовать «укажите площадку / добавьте WB» даже в завершённой задаче. Владелец: «агент не помогает, а только увеличивает нагрузку».
- **Root cause:** механика офферов (12.07): после оффер-комментария агента любой комментарий ответственного/постановщика БЕЗ имени агента роутился оферившему агенту, оффер оставался `state='open'` бессрочно (закрывался только явным «нет»). Сам оффер заканчивался «Ответьте прямо здесь — я увижу ваше сообщение» — люди не знали, что обычная переписка триггерит агента.
- **Fix (решение владельца — вариант «только по @», Albery `f571824`):** оффер-фолбэк удалён из `_b24_handle_task_comment_event` — в задачах агент отвечает ТОЛЬКО когда его позвали по имени/@ (`_b24_task_pick_agent`; @-упоминание Битрикса `[USER=id]Имя[/USER]` матчится). Каждый оффер теперь завершается фиксированной строкой (`task_offers._summon_tail`, добавляется кодом, не LLM): «[b]Нужна помощь — позовите меня через @.[/b] Введите @, выберите „<имя агента>“ (или напишите „<имя>“) и что сделать… Пока меня не позвали по имени — я в вашу переписку не вмешиваюсь». Промпт оффера больше не просит «ответьте прямо здесь».
- **Проверено:** матчер-смоук — @-форма/имя/«@албери» матчятся, «сделала заявки…»/«да»/«нет» = тишина; рестарт при inflight=0, журнал чистый.
- **Заметка:** офферы, запощенные ДО фикса, ещё содержат старый призыв «ответьте здесь» — ответы на них теперь молчаливые; новые офферы (в т.ч. ежедневный обход task_checkin) идут с @-инструкцией. Бэкапы: `/root/albery-backups-20260713/*.pre-patch5`.

## 2026-07-13 — два хода сотрудников зависли на все 600 сек (Наталья #669, Александр #709)

- **Impact:** Наталья (описание регулярной задачи со ссылками на Google-таблицы, 11:39) и Александр («Пришли ссылку на папку о компании», 13:22) прождали по 10 минут и получили отказ с литеральными `\n` в тексте и ложным обвинением Google-таблиц. За 14 дней таких ходов было 12.
- **Root cause:** зависший stream к ChatGPT/Codex-бэкенду. В state.db у обеих сессий ровно ОДНО сообщение (вопрос) — модель не выдала ни одного события за 10 минут; ни одного MCP-вызова в журнале. Параллельные ходы в те же минуты работали → сбой одиночного запроса, а не бэкенда целиком. Дефолтный per-call timeout hermes = 1800с; stale-детектор (90с) покрывает только «тишину до первого байта» (известная болячка codex silent-hang, hermes-agent #21444) — стрим, замолчавший ПОСЛЕ старта, висел бесконечно, и b24-кап 600с срабатывал первым. Ретрая на таймаут в b24bot нет (только на быстрый фейл).
- **Fix:** `/root/.hermes/config.yaml` → `providers.openai-codex.request_timeout_seconds: 240` (бэкап `config.yaml.bak-stall-timeout-20260713`). Для стримового Codex Responses это лимит ПАУЗЫ между событиями стрима: мёртвый стрим умирает за 240с → SDK кидает ReadTimeout → hermes ретраит вызов ВНУТРИ хода (`api_max_retries: 2`), не теряя уже сделанные tool-вызовы; живые длинные генерации (договоры юриста по 400-600с) шлют события и не задеты. Проверено: `get_provider_request_timeout('openai-codex', 'gpt-5.6-terra') = 240.0`, живой ход отвечает. CLI-ходы (все битрикс-агенты) читают конфиг на каждый запуск; hermes-gateway перезапущен для обновления кэша.
- **Заодно:** тексты таймаут-сообщений b24bot (Albery `4b818b4`) — реальные переносы строк вместо литеральных `\n`, честное «чаще всего это разовый сбой — отправьте ещё раз» вместо обвинения таблиц.
- **Осталось как защита:** жёсткий кап 600с в b24bot + ops-алерт «таймаут хода» в TG.
- **Как диагностировать такое:** state.db → sessions по времени старта (`id = YYYYMMDD_HHMMSS_*` МСК) → messages: одна user-запись без продолжения = зависший первый LLM-вызов; «hermes timed out after 600s» в журнале albery; mcp_tool_call-записей нет.

## 2026-07-12 — «Полина»: агент назвал ответственным человека, которого нет в компании

Проверено боем: «кто такая Полина» → «в актуальной оргструктуре такой нет».
- **Признаки порталов** (для будущих чисток): старый — `bitrix_user_id >= 100` (5-значные) и `bitrix_chat_id >= 10000`; текущий — id пользователей 1..62.
