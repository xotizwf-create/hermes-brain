## 2026-07-15 — агент «видит» только первый файл; файл без подписи теряется (личные сообщения)

- **Impact:** владелец загружал документы «Новинки»: агент загрузил один файл, следующие «не видит»; на новый файл ответил «этот договор уже загружен» (сослался на СТАРЫЙ), на «я прислал другой файл» — «мне передан только один файл». Файл без подписи не обрабатывался. Владелец: «раз и навсегда исправить, во всех агентах и везде, чтобы агент отвечал на всё, что кидают».
- **Root cause:** `_b24_message_extras` (личные сообщения) читал вложения ТОЛЬКО из встроенного блока события `data[PARAMS][FILES][<id>]`. Bitrix доставляет его НЕСТАБИЛЬНО — файл с подписью/следующий часто приходит БЕЗ встроенного блока (только по ссылке). Извлечение пустое → фолбэк `if not (msg_attachments or doc_blocks or image_texts): recent_atts/recent_doc`, и `_b24_compose_user_text` инжектил ПРЕДЫДУЩИЙ файл («если он пишет „этот документ“ — речь именно о них») → «уже загружен». Проверено: `im.dialog.messages.get(DIALOG_ID='14')` токеном приложения → 31 сообщение, 9 с `params.FILE_ID` — файлы доступны по ссылке, просто не в inline-блоке.
- **Fix (Albery `5aebb36`, Bitrix 1512):** `_b24_message_file_ids` восстанавливает id файлов сообщения через `im.dialog.messages.get` (params.FILE_ID) и мёрджит с inline по disk-id; имя — `_b24_disk_file_meta` (disk.file.get); скачивание токеном бота (участник диалога); ЛЮБОЙ не-image тип → путь документа (скачан+сохранён+подтверждён, не отбрасывается по расширению); `[:5]`→`[:10]`. Общий код → главный и ВСЕ суб-агенты; путь задач уже собирал все файлы.
- **Проверено (на реальных сообщениях):** `_b24_message_extras` c payload без inline-блока (только MESSAGE_ID) для dialog 14 → восстановил файл 2370, OCR 2125 симв., вложение создано. Полная цепочка: восстановленный файл → compose → `hermes_brain_answer` → агент загрузил ИМЕННО его в «Новинки» и дал ссылку. Рестарт при inflight=0, журнал чист.
- **Урок:** НЕ доверять inline-доставке вложений Bitrix (ни в DM, ни в задачах) — всегда восстанавливать по ссылке (FILE_ID). «Отвечать на всё» = не отбрасывать файл по типу, а хотя бы сохранить и подтвердить.

## 2026-07-15 — агент «не видит вложение» в задаче: 403 на файлах у технического юзера-не-участника

- **Impact:** Наталья Горюнова позвала агента в комментарии задачи №594 «Дни рождения сотрудников», приложив к тому же комментарию список сотрудников с датами — агент ответил «Не удалось скачать вложенный файл». Владелец: «АГЕНТ ИЗ ЛЮБЫХ ИСТОЧНИКОВ ДОЛЖЕН ВИДЕТЬ ЛЮБЫЕ ВЛОЖЕНИЯ, МЫ ЭТО УЖЕ ДЕЛАЛИ». Задача Александра на фикс — №1488.
- **Root cause:** НЕ пробел в коде чтения (файл был на упоминающем комментарии, и код его читал). Битрикс защищает файлы задач и комментариев по УЧАСТИЮ в задаче. Скачивание идёт под техническим пользователем вебхука (id 22 «ИИ Агент»), который НЕ участник большинства задач → `disk.file.get` возвращает **HTTP 403 ACCESS_DENIED** (подтверждено в журнале сервиса на 12:27:45 для файла 2394). И вебхук, и app-токен ходят как user 22 — оба 403. Заработало только после того, как 22 стал наблюдателем 594 (сообщение в задаче «ИИ Агент стал наблюдателем» → далее скачивание успешно). Формат: `UF_TASK_WEBDAV_FILES` = `["n<disk_file_id>"]` (снять «n»); файлы комментариев = `params.FILE_ID` сообщений чата задачи.
- **Fix (Albery `b1cda4e`, Bitrix 1488):** (1) `task_comment_files` само-восстанавливается: при пустом скачивании один раз делает вебхук-юзера АУДИТОРОМ (наблюдателем) задачи через `tasks.task.update AUDITORS` (идемпотентно — без записи и уведомления, если он уже ответственный/соисполнитель/аудитор/постановщик) и повторяет скачивание. Чинит все пути чтения разом (упоминание + `get_task_comments`). (2) `_b24_task_all_file_ids` собирает id файлов со ВСЕЙ задачи (тело `UF_TASK_WEBDAV_FILES` + `params.FILE_ID` всех комментариев чата), обработчик упоминания инжектит до 6 файлов «из любых источников», а не только из упоминающего комментария. Право технического юзера добавлять наблюдателя проверено на одноразовой задаче (create→update AUDITORS→delete).
- **Проверено (без изменения реальных участников, без уведомлений):** сбор по 594 находит 2394; OCR читает таблицу «Имя и фамилия | Дата рождения | Месяц рождения»; детерминированная симуляция 403 — скачивание не проходит, пока не вызван `_b24_ensure_task_auditor`, после «выдачи доступа» повтор читает файл (821 симв.). Рестарт при inflight=0, журнал чист.
- **Урок:** «агент не видит вложение» ≠ всегда баг чтения. Если объект Битрикса (файл, диск, задача) отдаёт 403 — проверь ДОСТУП технического юзера: интеграционный вебхук-юзер видит файлы задачи только как её участник. Доступ к объекту даётся участием/шарингом, и это надо обеспечивать в коде, а не полагаться на ручное «стал наблюдателем».

## 2026-07-15 — мониторинг был слеп: таймаут/перегрузка/ошибка хода писались как «ок» (аудит F09)

- **Impact:** страница здоровья Центра Агента, бейджи ошибок в диалогах и почасовой самоконтроль (`self-check`) показывали ЗЕЛЁНОЕ даже когда ходы реально падали по таймауту (600с) или упирались в перегрузку модели. Из аудита системы (`tmp/albery-agent-system-audit-2026-07-09_2026-07-15.md`, F09). Факт: журнал ходов за 7 дней = 348 `ok` / 0 `timeout`, при том что реальные таймауты и ответы «Our servers are currently overloaded» в journalctl были.
- **Root cause:** ход завершается 4 исходами — успех, таймаут, «занят» (кап одновременных), ошибка модели. Но `hermes_brain_answer` во ВСЕХ случаях возвращает строку пользователю (в т.ч. вежливую заглушку «Не успел за 600 c»). Вызывающий `_b24_app_process` инициализировал `status='ok'` и менял его на `error` только в `except`, так что таймаут/занят/сентинел-ошибка (возвращаемые как обычные строки) писались в `bitrix_bot_interactions` со статусом `ok`. Плюс UI жёстко ставил `"last_status":"ok"` (список диалогов) и `"status":"ok"` (ветка переписки).
- **Fix (Albery `f276101`, Bitrix 1484, бэкапы `b24bot.py.honest-*`, `agent_center.py.honest-*`):** потоко-локальный «итог хода» `_TURN_OUTCOME` + `_turn_outcome_reset/set/get`. Каждый return в `hermes_brain_answer` проставляет настоящий статус (cancelled/busy/timeout/error/ok); вызывающий берёт его через `_turn_outcome_get()` вместо константы. Оба пути логирования (приватный чат + упоминание-в-задаче) честны. В agent_center: `_dialog_last_status()` даёт список диалогов статус последнего реального хода; ветка помечает исходящее сообщение статусом сбоя, если рядом (≤120с) есть провал в журнале. Аддитивно, управляющий поток на успехе не меняется, фронт не пересобирали (изменились только значения в JSON).
- **Проверено:** смоделированный таймаут через реальную функцию → строка журнала `('timeout','hard timeout 600s')`, текст пользователю — прежняя заглушка; busy→'busy'; реальный успешный ход (мозг ответил «привет») → 'ok' (регрессии нет); `_dialog_last_status()` вернул 43 записи; новый SQL ветки валиден на живых данных. Рестарт при inflight=0, сервис активен (HTTP 302), журнал чист.
- **Урок:** мониторинг обязан строиться на честном источнике. Если функция возвращает user-facing строку на любом исходе, вызывающий не отличит успех от заглушки — исход надо передавать отдельным каналом, а не угадывать по тексту. Мониторинг уже считал «статус ≠ ok» — достаточно было перестать врать в источнике.

## 2026-07-14 — юрист «снова делает договоры фигово» + литеральные <b> в TG

- **Impact:** договоры юриста потеряли ГОСТ-оформление (интервал 1,15 вместо 1,5, нет выравнивания по ширине, случайная структура); владелец: «стрем полнейший, может мы перемудрили с ограничениями?». Плюс в TG-ответах литеральные `<b>`/`[b]`.

Юрист не видел инструкцию по оформлению. Обрезание — молча, ни строчки в логах. «Раньше классно» = скилл был короче лимита.
  2. **Вёрстка отдана модели**: docformat по замыслу «no app-side opinions» — justify/интервал/отступы зависели от HTML модели; смена модели на Terra (12.07) сместила стиль HTML.
  3. **TG-бот шлёт plain text без parse_mode** — разметка модели уходила людям литерально.
- **Fix (Albery `e5ea5b5`, Bitrix 1454):** лимит 8k→24k + **WARNING при любом обрезании** (тихих обрезаний больше нет); docformat гарантирует ГОСТ-инварианты кодом (justify для обычных `<p>` по умолчанию, интервал 1,5 по умолчанию; явные стили модели сильнее; заголовки/списки/таблицы не тронуты) + дефолт 1,5 в export_document; tg_agent вычищает `[b]`/`<b>`/`**` в send_text.
- **Проверено:** голый HTML без стилей → ГОСТ-docx (поля 3/1,5/2/2, TNR, 1,5, центр/justify); живой ход юриста (48с) → 29 абзацев все 1,5, justify, шапка центром, реквизиты таблицей, TNR всюду.
- **Урок 1:** «агент внезапно стал тупее» при живой инфраструктуре = ищи ТИХИЙ лимит/обрезание в конвейере промпта, а не «перемудрёные ограничения». Любой cap обязан логировать срабатывание.
- **Урок 2:** инварианты качества (ГОСТ-вёрстка) держать в КОДЕ; модели оставлять содержание. Это же уже было с ценами WB (история ≠ витрина) и офферами (контекст задачи).

## 2026-07-14 — цены WB находились для 14 из 43 (моя же защита рубила годные)

- **Impact:** «Пройдись по всем артикулам» → агент нашёл цены только по 14 из 43, по остальным «WB ограничил» — хотя цены у них ЕСТЬ. Владелец: «что-то сломалось, сделай как у Claude/ChatGPT, чтобы всё находило».
- **Root cause:** вчерашняя МОЯ защита оказалась слишком грубой. search.wb.ru рейт-лимитит по IP (429); мой код после нескольких запросов взводил ГЛОБАЛЬНЫЙ кулдаун, зануляя весь остаток батча (wb_throttled). Диагностика: 14 ok, 24 «throttled» (цены есть!), 2 реально пустых. card.wb.ru и каталог продавца мертвы (404/403) — рабочий источник один: search.wb.ru.
- **Fix (Albery `25bac1a`, Bitrix 1434) — «умно и терпеливо», как настоящие системы:** (1) **позитивный кэш 45 мин** — цену КАЖДОГО товара из ответа поиска запоминаем (поиск = каталожная страница; соседи оцениваются даром), прогресс копится между вызовами, не сбрасывается; (2) **два IP round-robin** (прямой RU + VPN = независимые квоты 429); (3) **убран глобальный кулдаун** — на 429 короткий бэкофф и дальше; (4) честный статус: `card_hidden` (WB реально не отдаёт карточку — нет в наличии, финал) vs `no_price/skipped` (троттл — агент повторяет вызов, кэш ускоряет, пока не останутся только card_hidden). Правило в Маршрутной карте обновлено.
- **Проверено:** симуляция 43 живых → **39/43 цен + 4 честно недоступны + 0 зависших за 120с** (было 14); живой ход агента на 12 → 9 цен + 3 «карточка скрыта WB» за 85с; таблица Софьи перезаполнена (39 цен, 4 пустых). 4 недоступных (355600658, 455277533, 920423972, 969587953) реально отсутствуют в выдаче WB по любому запросу.
- **Урок:** защита от предыдущей проблемы (залипание/таймаут) не должна создавать новую (потеря годных данных). Кэшируй результаты (в т.ч. побочные — соседей из ответа), чередуй ресурсы, не зануляй партию целиком; «недоступно» помечай только когда источник ПОДТВЕРЖДЁННО молчит, а не по своему лимиту.

- **Impact:** владелец: «Пройдись в таблице по всем артикулам, заполни цены» → ход умер по таймауту 600с (#760); «следующие 10» → 404с (~40с/артикул). Требование: агент не должен тупить на одном элементе, если остальные тянутся; недоступное — пропускать и честно называть.

  3. **Поиск WB: прямой RU-IP первым** (отдельная 429-квота), VPN — фолбэк; устойчивый 429 → глобальный кулдаун 120с + честный статус `wb_throttled` («повторите через 3-5 минут»).
  4. **Маршрутная карта (hot, все агенты):** «НЕ ЗАЛИПАЙ НА ОДНОМ ЭЛЕМЕНТЕ» — для любых списков: макс. одна повторная попытка → пропусти → в конце честно перечисли с причиной; частичный результат ценнее нулевого; массовые цены WB → get_wb_prices пачками.
- **Проверено:** батч 5 (2 мёртвых) = 29с; e2e живым ходом список из 8 (2 мёртвых) = **55с**, цены собраны, «Недоступны: … WB сейчас не отдаёт цену» (раньше — смерть хода на 600с).
- **Урок:** «пропусти и иди дальше» нельзя оставлять на совести модели — бюджет времени, кэш промахов и батч-API должны быть в инструменте; промпт лишь договаривает честную подачу.
