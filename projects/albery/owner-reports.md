# Albery — отчёты собственнику и слой AI-инструкций

Как Albery-агент формирует управленческие отчёты собственнику и где живут правила, которые
этим управляют. Это знание о **продакшене Albery**, а не о репозитории мозга.

## Где живут инструкции (важно: это и есть прод)
- Правила поведения агента и контракты отчётов хранятся в БД Albery, раздел
  `Настройки → Инструкции для ИИ` (таблица `ai_instruction_folders`), и читаются агентом при
  каждом старте через MCP-инструмент `start_here_always_read_ai_instructions`.
- Редактируются эти правила через MCP `upsert_ai_instruction(path, content)`. **Стейджинга нет:**
  правка через `upsert_ai_instruction` — это запись напрямую в боевую БД, изменение действует сразу.
- ⚠️ В отличие от отчётов (versioned, `is_current`), инструкции в БД **не версионируются** —
  только `updated_at`, истории редакций нет. Перед большой правкой имеет смысл сохранить текстовую
  копию текущей инструкции.

## Типы отчётов и их контракты
- **Ежедневный отчёт по компании** — инструкция `Формирование отчетов / Ежедневный отчет по компании`;
  основной контракт структуры — активный промпт в `Сводная аналитика → Настройка промтов`.
- **Zoom-созвоны** — инструкция `Формирование отчетов / Зум-созвоны`; JSON-контракт — промпт
  `zoom_processing` (`get_report_contract category_key=zoom_processing`). В нём же блок
  `weekly_review` для пятничной контрольной встречи (Наталья/Артур: сделано/не завершено/цели с весами KPI).
- **Еженедельный отчёт собственнику** — инструкция `Формирование отчетов / Еженедельный отчет по
  компании` (создана 2026-06-05). Отдельного backend-контракта нет
  (`get_report_contract category_key=owner_weekly` → пусто), сама инструкция и есть контракт.
  Сохраняется через `save_owner_weekly_report` (таблица `owner_weekly_reports`, versioned).

## Контракт еженедельного отчёта собственнику (суть)
Управленческая записка для приёмки, обязательная структура: (1) общий вердикт по всей команде →
(2) динамика vs прошлая неделя + цифры недели → (3) ритм встреч (таблица день×норма×факт×статус) →
(4) соответствие регламентам (матрица решений, ФИО, веса KPI, отчётность, постановка, фиксация) →
(5) поведенческие факторы с эпизодами/таймкодами → (6) **пофамильная оценка руководителей**
(Наталья, Артур) — **оформляется ТАБЛИЦЕЙ по каждому руководителю** (`задача (план+вес) | факт
(таблица ↔ созвон) | вердикт ✅принять/⚠️частично/🚫заблокировано/❌не принимать | артефакт`),
строка на каждую задачу недели → (7) несостыковки «созвон ↔ Google-таблица» → (8) план следующей
недели → (9) решения собственника (действие/ответственный/срок) → (10) итог.

Рабочая неделя для отчёта — **пн–чт** (раздел 3): **пятница — выходной**, поэтому период отчёта
пн–чт, а контрольная встреча итогов недели проводится в **четверг**.

Источники: дневные owner-отчёты, Zoom-отчёты (особенно пятничный `weekly_review`), лист «Задачи
недели» из Google-таблицы «Операционная встреча. Албери 2.0», Bitrix-задачи, чат руководителей,
регламенты из «О компании», предыдущий недельный отчёт (для динамики).

## Жёсткие правила, выстраданные на практике
1. **Идентификация людей — по транскрибации, а не по метаданным.** Кто присутствовал и под каким
   именем — только по подписям спикеров в расшифровке Zoom. Техническое имя Zoom-аккаунта / тема
   созвона («Зал персональной конференции Координатор») — это служебные метаданные, НЕ имя участника.
   Запрещено выводить «участник не под своим ФИО» из метаданных — это привело к ложному обвинению
   Натальи 2026-06-05.
2. **Собственника (Александра Никитенко) нельзя ставить ответственным** в таблице решений — он
   принимает решения, но не исполняет их и не «внедряет» операционные процессы. Исполнители —
   Евгений Палей (ген. директор, арбитр по матрице решений) и руководители (Наталья, Артур, Дмитрий).
3. **«О компании» читать по именам файлов, а не многословным keyword-поиском.** `search_company_knowledge`
   по фразе часто возвращает пусто, даже когда документ есть; открывать через `list_company_files` +
   `get_company_file`. Никогда не делать вывод «документа нет» по одной неудачной попытке поиска.
4. **«Выполнено» из таблицы не принимать без проверки по созвону и без артефакта.** При открытом
   блокере статус — «частично / заблокировано».
5. **Пятница — выходной; контрольная встреча итогов недели — в четверг.** НИКОГДА не помечать
   «контрольная встреча в пятницу не проведена» как нарушение — её там и не должно быть. Итоговую
   встречу оценивать по четвергу; пятницу в ритме показывать как выходной.
6. **Аномалия Битрикса = подсветить причину, а не выдавать устаревшие цифры.** Перед цифрами по
   задачам проверять свежесть синхронизации. Если новых задач за неделю нет / появление прекратилось —
   наиболее вероятная причина: **истекла подписка Bitrix Marketplace** (отрезан REST → задачи не
   подтягиваются), и текущие числа («открыто 78» и т.п.) УСТАРЕЛИ. В этом случае не подавать их как
   актуальные, а писать «⚠️ Битрикс: новых задач нет — вероятно истекла подписка Маркетплейс, цифры
   устарели» + вынести отдельным риском и решением собственнику. Общее правило — всегда работать с
   актуальными данными, не выдавать старые за свежие. См. [[albery-bitrix-rest-subscription]].
