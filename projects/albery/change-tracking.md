# Albery — каждое изменение = закрытая задача в Битриксе (инженерная дисциплина)

Стоячее требование владельца (2026-07-13): **любое изменение в работающей системе Albery
ведётся как отдельная задача в Битрикс24 — с полным контекстом, ссылкой на коммит и временем,
чтобы был виден хронологический порядок.** Работаем как инженеры топовых компаний: каждое
изменение прослеживаемо и к нему можно вернуться. Это НЕ замена changelog'у мозга — это
дополнительный, видимый команде журнал в самом Битриксе.

## Когда заводить задачу

Одна задача = **одно атомарное изменение = один коммит** в репозитории Albery
(`github.com/xotizwf-create/Albery`). Заводить при изменении **работающей системы**:
код (`b24bot.py`, `mcp/context_server.py`, модули), конфиг (`/root/.hermes/config.yaml`, `.env`,
systemd), инструкции агента в БД (Маршрутная карта и пр.), `SOUL.md`, промпты, миграции.

Не нужно заводить задачу на чисто документационные правки мозга (`projects/albery/*.md`,
память) — они и так есть в `logs/changelog.md`. Но если правка инструкций **меняет поведение
агента на проде** (например, Маршрутная карта, SOUL) — это изменение системы, задача нужна.

## Поля задачи (шаблон)

| Поле | Значение |
|---|---|
| **Ответственный** | Александр Никитенко, `bitrix_user_id=16` |
| **Постановщик (creator)** | ИИ Агент, `bitrix_user_id=22` |
| **Дедлайн** | момент выполнения работы (МСК) — задаёт хронологию; несколько изменений подряд идут с возрастающим временем |
| **Название** | `Albery: <суть изменения одной строкой>` |
| **Описание** | контекст: что за проблема, КТО столкнулся/сообщил, симптом, первопричина (root cause) |
| **Критерий результата** (`result_criteria`, обязателен) | что считается выполненным |
| **Комментарий-результат** (при закрытии) | что именно сделано + коммит + проверка + откат |
| **Статус** | сразу закрыта (работа уже сделана и проверена) |

## Что значит «полный контекст» (чек-лист)

Описание + комментарий-результат вместе должны содержать ВСЁ, чтобы через полгода понять
изменение без раскопок:

1. **Проблема и кто столкнулся** — симптом словами пользователя, кто сообщил (Софья/Наталья/…),
   когда, номер диалога/задачи если есть.
2. **Root cause** — настоящая причина, не пересказ симптома.
3. **Что сделано** — конкретные действия, затронутые файлы/модули.
4. **Ссылка на коммит** — хеш + полная ссылка:
   `https://github.com/xotizwf-create/Albery/commit/<hash>`. Если правка не в git
   (конфиг/БД-инструкция) — указать файл и бэкап.
5. **Как проверено** — что за проверка боем/тест, какой результат (цитата вывода).
6. **Откат** — путь к бэкапу (`/root/albery-backups-YYYYMMDD/…`, `*.bak-…`) или `git revert <hash>`.
7. **Связь с другими** — если изменение продолжает серию, сослаться на предыдущую задачу.

## Хронология

- Задачи создаются **в том порядке, в каком делались изменения**; дедлайн = время работы, поэтому
  сортировка по сроку/дате создания сразу даёт правильную последовательность.
- Серия изменений за сессию — отдельные задачи, каждая со своим коммитом и временем; в описании
  можно указать «шаг N сессии от <дата>».

## Как создавать (технически, проверено 2026-07-13)

Задачи создаются напрямую вызовом MCP-инструментов из отдельного скрипта на проде (импорт
`mcp.context_server` подгружает `.env`). Полный рабочий скаффолд был в этой сессии; ключевое:

```python
import os, sys
os.environ["B24_TASK_OFFER"] = "0"           # ВАЖНО: глушим оффер-комментарий на этих готовых задачах
sys.path.insert(0, "/var/www/albery")
from mcp import context_server as cs

created = cs.tool_create_bitrix_task({
    "title": "Albery: …", "description": "…",
    "responsible_bitrix_user_id": 16, "creator_bitrix_user_id": 22,
    "deadline": "2026-07-13T18:00:00+03:00",   # время работы
    "result_criteria": "…",
})
tid = created["task_id"]
cs.tool_complete_bitrix_task({
    "bitrix_task_id": tid, "on_behalf_bitrix_user_id": 22,
    "expected_title": "Albery: …",
    "result_text": "СДЕЛАНО (Albery <hash>):\n- …\nПроверка: …\nОткат: …",
})
```

### Грабли и правила (набито в этой сессии)

- **Глушить оффер:** `B24_TASK_OFFER=0` до вызова — иначе на готовую задачу прилетит
  комментарий-предложение «могу помочь». Для этих журнальных задач он не нужен.
- **Проверять результат ТОЛЬКО через `cs.tool_get_task_comments`** (`include_service=False`).
  Сырой REST `task.commentitem.getlist` на этом портале отдаёт **ложный ноль** — не верить ему.
  У правильно оформленной задачи: `human_comments=1` (результат), `service_comments=2` (создание+закрытие).
- **Комментарии на этом портале НЕ удаляются и НЕ редактируются через REST** — и вебхуком, и
  app-токеном возвращается `ACTION_NOT_ALLOWED`. Если в задачу попал лишний/тестовый комментарий:
  **удалить всю задачу целиком** (`tasks.task.delete`, работает через вебхук) и создать заново
  начисто. Поэтому: не «тестировать» на боевой задаче — сначала отладить скрипт, потом один
  чистый прогон.
- `result_criteria` обязателен (инструмент откажет без него). Дедлайн в прошлом требует
  `confirm_past_deadline=true` — при записи «задним числом» указывать реальное время работы; если
  оно уже прошло, передать этот флаг.
- Закрытие с `result_text` само постит комментарий-результат с префиксом «✅ РЕЗУЛЬТАТ ЗАДАЧИ:»
  и закрывает задачу (`STATUS=5`, `STATUS_CHANGED_BY=22`).

Порядок и задачи этой сессии (все на Александра 16, дедлайн 13.07 18:00, закрыты):

2. `1374` — агент пересобирал присланные документы вместо точечной правки — Albery `c226768`.
3. `1376` — ходы агента зависали на 10 минут (таймаут) — Albery `4b818b4`.

удалить — удалил задачу целиком и создал заново. Отсюда правило «не тестировать на боевой».)
