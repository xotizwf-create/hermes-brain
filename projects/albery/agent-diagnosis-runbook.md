# Albery — ранбук диагностики «агент тупит» + таксономия сбоев

Выстрадано 16.07.2026 (двойная сессия: кнопка Apps Script + потеря контекста) и ретроспективой
всего корпуса: **980 переписок** (15.06–16.07), **124 correction-хода** (12%), **47 инцидент-задач**
инженерного журнала. Когда владелец говорит «агент тупит / теряет контекст / не может сделать» —
идти по этому ранбуку, НЕ гадать.

## Порядок диагностики (обязательный)

1. **Дамп диалога.** `bitrix_bot_interactions` по dialog_id/периоду (полные Q+A, не превью):
   психологию сбоя видно только по полному тексту ответов агента — где он врёт «готово»,
   где зациклился, где потерял нить. Полезно: `\pset format unaligned` + fieldsep.
2. **Состояние сессии.** `bitrix_bot_sessions`: `epoch` (большой = частые сбросы), `history_floor_id`
   (что скрыто), `summary` (пустой = нет переносимого контекста), ключ scope = `slug:dialog_id`.
   С 16.07: сброс только после **3 ч** простоя, полный, с уведомлением пользователю
   (`_b24_notify_session_reset`, от бота того же агента). Ручная «Новая сессия» — жёсткий сброс.
3. **Журнал.** `journalctl -u albery --since ... | grep -iE 'error|rejected|timeout'` — искать
   ПОВТОРЫ одной ошибки (агент долбит мёртвый вызов), 404 с искажёнными ID, 429 троттлинг.
4. **Слой инструментов — ГЛАВНЫЙ подозреваемый при повторных провалах.** Вопрос не «почему модель
   не смогла», а «мог ли инструмент это сделать В ПРИНЦИПЕ». Три инцидента 16.07 были неисполнимы
   на уровне тулов (create без parentId; scripts.run 404 by design; нет чтения значений). Проверять:
   сигнатуру функции в `app.py`, схему в `mcp/context_server.py`, whitelist (`agents.tools` в БД,
   `CORE_TOOL_NAMES`/`OPS_TOOL_NAMES`).
5. **Чинить на правильном слое, в порядке:** инструмент (возможность + честная ошибка) →
   инструкция (реестр `agent_knowledge/instructions/`, universal = всем; «Маршрутная карта» и
   «Базовое поведение» инжектятся каждый ход, кап 30k, сейчас ~17.6k) → промпт b24bot.
6. **Верификация = эталон из диалога.** Найти в переписке числовой пример пользователя и прогнать
   именно его (16.07: SKU 969588068 → июнь 488160/1357/360/23). Синтетика для лайфцикла
   (fake-строки сессий), стабы для отправок (перехват `_b24_app_call`).
7. **Деплой:** py_compile → рестарт только при `bitrix_inflight_turns == 0` → коммит+push (прод
   пушит сам) → закрытая Bitrix-задача (правило №8, `change-tracking.md`; результат проверять по
   `comment_id` из ответа, НЕ по ридерам) → changelog мозга + память.

## Антипаттерны исправлений (мои собственные грабли)

- **Не изобретать «умную память», когда владелец просит простую механику.** Мягкий сброс с
  дайджестом (`177fdbd`) владелец отверг: «никаких дайджестов». Правильный дизайн: простое
  правило + ЯВНОЕ уведомление пользователю + путь возврата («Ответить»). Спросить дизайн, если
  меняешь видимое поведение.
- **Не чинить промпт, когда сломан инструмент.** Сначала п.4 ранбука.
- **Не верить ридерам без записи-подтверждения:** комментарии задач — только `comment_id` из
  ответа инструмента; ридеры (`get_task_comments`, raw getlist) врут нулём.
- **Никогда не ставить model-вызов (`_b24_summarize_segment`, hermes -z) на критический путь
  ответа** — до 90с задержки на первый ход.
- **Тестовый комментарий на боевой задаче = пересоздание задачи** (комментарии не удаляются).
