## Bitrix webhook: сначала быстрый ACK, потом всё сетевое (2026-07-04)
Инцидент: во время сетевой деградации/похожей на DDoS ситуации Bitrix дважды оборвал `POST /bitrix/imbot/app` как `499`, хотя сервер Albery был живой. Причина: до HTTP-ответа webhook синхронно делал косметические вызовы в Bitrix API (`eyes` reaction + typing); при тормозах Bitrix/маршрута эти вызовы могли зависнуть на сетевом таймауте, и Bitrix считал бота неработающим. Исправление в `b24bot.py`: входящий текстовый webhook должен **сразу** стартовать background worker и вернуть `accepted`; reaction/typing и тяжёлый ответ агента выполняются уже внутри worker. Правило на будущее: до ответа Bitrix нельзя делать внешние REST-вызовы, скачивание файлов, OCR или другие медленные операции; максимум — локальная валидация и запуск фонового потока. Проверка после hotfix: `/bitrix/imbot/app` снаружи отвечал за ~0.01–0.03с, сервис активен, ошибок после рестарта нет.

## Доступ к Универсальному агенту = СТРОГИЙ allowlist (2026-07-03)
Основной (универсальный) бот теперь отвечает ТОЛЬКО своей «Команде и доступам» = строкам
`agent_access` с тиром ≠ `none`. Кого нет в списке (нет строки ИЛИ строка `none`) — бот НЕ
отвечает, отдаёт системную плашку «нет доступа». Список команды = список доступа (правило
владельца). Реализация `b24bot._b24_main_allows(uid)`: `uid ∈ _b24_owner_user_ids()` (bootstrap-
владелец всегда допущен, чтобы не залочиться) ИЛИ `agent_access[uid] ∉ {None, 'none'}`. Оба гейта
отказа (сообщения `denied` и команды `cmd_denied`) для основного бота (`agent is None`) переведены
с «`== none`» на `not _b24_main_allows(...)`. ⚠️ ПОВЕДЕНЧЕСКОЕ ИЗМЕНЕНИЕ: раньше юзер без строки =
`faq` (бот отвечал); теперь без явного гранта = отказ. На момент внедрения доступ к основному боту
имели ровно 4 человека (id 14 Евгений Палей, 16 Александр Никитенко, 26 Сергей Виноградов,
28 Артур Степанян); остальные (9 строк `none` + все без строки) — отказ. Кому нужен доступ —
добавить в «Команду» (грант `ops`). Гейт субагентов (`_agent_allows` по `agent_members`) не менялся.

## Имя + должность агента, синхронизация с Bitrix (2026-07-03)
У агента есть `name` и `position` (должность). Миграция `040_agent_position.sql` — колонка
`agents.position` (дефолт «ИИ-агент Albery»). При создании/сохранении и имя, и должность
пушатся в Bitrix-бота: `_sync_bitrix_bot(bot_id, name?, position?)` → `imbot.update` с
`FIELDS.PROPERTIES.{NAME, WORK_POSITION}` (обобщён из старого `_rename_bitrix_bot`). Приложение
= единственный источник правды, в мессенджере ровно то же. Поля «Имя агента» + «Должность» в
создании и в редакторе (у основного тоже — пушит в главного бота). Имя основного (универсального)
агента САМОСИНХРОНИЗИРУЕТСЯ из Bitrix: detail-эндпоинт читает живое имя бота и адоптит в строку
(источник правды для пред-существующего главного бота), поэтому в приложении ровно то же, что в
мессенджере («Агент Албери», bot_id 24). ⚠️ Боты НЕ возвращаются `user.get` — имя берётся из
`imbot.bot.list` (по `ID` бота из state). Для основного detail отдаёт РЕАЛЬНЫЙ bot_id (24) для
показа, поэтому плашка «зарегистрировать бота» не появляется; при этом строка `main` продолжает
хранить `bitrix_bot_id=NULL` (чтобы `agent_for_bot_id` не резолвил главного бота как субагента).

## Управление агентской системой из чата — owner-only MCP-инструменты (2026-07-03)
Топ-доступный агент может рулить всей системой прямо в переписке. 6 инструментов (класс `admin`,
подтверждение при включении, вне ops/faq-коннекторов; в `context_server.AGENT_MGMT_TOOL_SPECS`,

`create_agent`, `update_agent` (имя/должность/промпт/вкл-выкл), `delete_agent`, `set_agent_tools`
(вкл/выкл инструменты по именам), `set_agent_team` (add/remove по имени или id; для `main` — правит
`agent_access`, для субагента — `agent_members`). Реализация: хендлеры оборачивают те же
`/api/agent-center/*` эндпоинты через ВЛОЖЕННЫЙ `app.test_request_context` → валидация/энфорсмент
идентичны UI (без дублирования логики); имена сотрудников резолвятся `agent_center.resolve_bitrix_user`
(подстрока по директории портала). ⚠️ Гейт = кто в команде агента, у которого эти тулы включены:
владелец включает их через панель «Возможности» ТОЛЬКО на агенте с узкой доверенной командой
(caller-id до MCP-тула не доходит, поэтому безопасность = ограниченная команда). На /mcp (admin)
теперь 79 инструментов (73 + 6), /mcp-ops и /mcp-faq не изменились. Смоук: create/update/list/
set_tools/set_team(add «Александр»→id16)/delete отработали на проде.

## Единый редактор для ВСЕХ агентов (2026-07-03)
Один компонент `AgentEditor` (фронт `Интерфейс/src/agent/views/AgentsView.tsx`) рендерит КАЖДОГО
агента — и Универсального (основного), и субагентов — с идентичным макетом: имя · каналы связи ·
роль/промпт · команда и доступы · личные инструкции и навыки · «Возможности агента» (инструменты +
инструкции/скиллы). Старый отдельный редактор основного (~560 строк) удалён. Отличия основного —
ТОЛЬКО в данных: его нельзя удалить (кнопка «Удалить» скрыта), а «Команда» = общий по компании
`agent_access` (детейл `is_main=true`, `_main_access_member_ids`; PATCH `main` реконсилит команду
в `agent_access`: добавленным — грант, убранным — удаление строки). У субагентов команда =
`agent_members`. Выбор редактора в `AgentsView` больше не ветвится по типу — всегда `AgentEditor`.

## Субагенты (главная механика)
Субагент = **отдельный Bitrix-бот + собственная capability-граница + личное самообучение**.
- **Создание из UI** (модалка «Создать агента»): имя, уровень (faq = только знания / ops = без
  админ-инструментов), роль-промпт, участники. POST `/api/agent-center/agents` делает всё сам:
  `imbot.register` через ТО ЖЕ локальное приложение (новое приложение не нужно; CODE
  `albery_agent_<slug>`) → бот появляется в мессенджере; в `/root/.hermes/config.yaml` секцию
  `mcp_servers` дописывается коннектор `agent-<slug>` (ТЕКСТОВАЯ вставка с бэкапом + yaml-валидация,
  комментарии конфига не трогаем; рестарт Hermes НЕ нужен — CLI читает конфиг на каждый ход).
- **Таблицы** (миграция `037_agents.sql`, идемпотентная, в ALWAYS_APPLY): `agents` (slug, name,
  role_prompt, tier, tools[], bitrix_bot_id, mcp_token, is_active), `agent_members`,
  `agent_instructions` (source: owner|self) + колонка `agent_slug` в `bitrix_bot_interactions`.
- **Роутинг в b24bot**: событие несёт BOT_ID → `agent_for_bot_id()` (кэш 60с); неизвестный id =
  основной агент. Гейт: список участников агента (пустой = все, кто не `none`); выключенный
  агент честно отвечает «выключен». Ход субагента: toolset `-t agent-<slug>`, в промпт — роль +
  накопленные инструкции + нудж на самообучение; interactions пишутся с `agent_slug`.
- **Изоляция структурная**: `/mcp-agent/<slug>/<token>` — единственный коннектор агента:
  тир-набор ∩ опц. whitelist `tools[]` (бэкенд умеет, UI пока нет) + 3 инструмента
  самообучения, обрабатываемых ПРЯМО в эндпоинте со slug из URL: `upsert_my_instruction` /
  `list_my_instructions` / `delete_my_instruction`. Агент физически не достаёт админ-инструменты
  и чужие/глобальные инструкции. Лимиты: 30 самоинструкций, 8k символов; инструкции владельца
  защищены от удаления агентом.
- **Имена = Bitrix в обе стороны**: основной агент показывает реальное имя бота с портала
  (user.get, кэш 10 мин; сейчас «Агент Албери», bot_id 24) и переименовывается через
  PATCH `/agents/main` → `imbot.update`; переименование субагента тоже пушится в Bitrix.
  Регистрация не удалась → жёлтая кнопка «зарегистрировать бота» (POST `/agents/<slug>/register-bot`).
- **Редактор субагента**: промпт/имя (сохранение), вкл/выкл, участники, личные инструкции
  (бейджи «сам выучил» 🤖 / «владелец» 💬, ручное добавление/удаление), удаление агента
  (unregister + чистка коннектора + каскад БД).

### Настройка возможностей субагента (2026-07-03) — тумблеры/галочки с РЕАЛЬНЫМ энфорсментом
Панель «Возможности агента» в редакторе субагента (`AgentCapabilityPanel`), эндпоинты
`GET/PUT /api/agent-center/agents/<slug>/config`. Миграция `038_agent_config.sql` (идемпотентная,
ALWAYS_APPLY): колонка `agents.tools_customized` + таблица `agent_knowledge_links (agent_id, kind
∈ instruction|skill, ref_id)`.
- **MCP-инструменты (тумблеры)** — энфорсмент НАСТОЯЩИЙ, на уровне коннектора: `_agent_tool_names`
  = выбранные инструменты ∩ `_agent_allowed_pool(agent)` ∪ **фиксированный baseline**
  `MANDATORY_AGENT_TOOLS` (start_here / get_context_guide / get_ai_instructions /
  search_company_knowledge). `tools/list` коннектора отдаёт ровно этот набор → выключенный
  инструмент невидим и невызываем. Дефолт (`tools_customized=false`) = пресет уровня
  (`_agent_preset_default`).
- **Уровень доступа УБРАН как отдельная настройка (2026-07-03, по требованию владельца).** Мощь
  агента = какие MCP-инструменты включены, и всё. `_agent_allowed_pool` = ПОЛНЫЙ реестр (73) для
  ЛЮБОГО агента; admin-инструменты (`upsert_ai_instruction`, `update_ai_capabilities`,
  `delete_bitrix_task`, `delete_zoom_call_report`, `get_agent_monitoring`) можно включить любому,
  но у них бейдж `class=admin` + подтверждение при включении в UI (это заменило гейт по уровню).
  Селектор уровня убран из создания и редактора; новый агент стартует с широкого пресета `ops`.
  Колонка `agents.tier` осталась в БД (миграция 039 не откатывалась, дефолт ops), но на гейтинг
  больше НЕ влияет — только сид пресета для не-кастомизированных. Смоук на проде: faq-агенту
  `delete_bitrix_task` теперь `allowed=true`, PUT сохраняет, коннектор отдаёт.
- **Инструкции и скиллы (галочки)** — из общей библиотеки (`ai_instruction_folders` + Hermes-скиллы
  из `_hermes_skills()`). Подключённые линкуются в `agent_knowledge_links`; на ходу
  `agent_selected_knowledge(agent)` инжектит в промпт субагента ТОЛЬКО выбранные (инструкции —
  полным текстом; скиллы — с жёстким «Пользуйся ТОЛЬКО этими навыками»). Не выбрано = не
  инжектится = агент не может применить.
- ⚠️ **Честная граница энфорсмента скиллов**: инструменты и инструкции держатся на движке
  (скоуп коннектора + инжект = у агента их физически нет/есть). СКИЛЛЫ Hermes грузит глобально и
  рекламирует модели на каждый CLI-ход — спрятать невыбранные на уровне движка без правки gateway
  нельзя. Поэтому скилл-селекция = промпт-энфорсмент. На практике большинство скиллов инертны без
  своих MCP-инструментов (а те гейтятся жёстко), так что это мягкий край, не дыра. Настоящее
  движковое скрытие скиллов — отдельная задача (правка Hermes gateway / per-run skills dir).

## Документы = оформление на стороне агента: export_document (2026-07-05)
Инцидент: ИИ-юрист провалил задание по договору — юзер просил «приложения с новых страниц,
без иероглифов, внизу без рекомендаций» (interaction 318, ход 573с → ошибка LLM), а до того
«внутри далеко не гост» (301). Корень (диагноз владельца подтвердился): оформление жило в
хардкод-экспортере приложения (`_b24_text_to_docx` парсит болтовню чата с [b]/эмодзи) — агент
физически не мог влиять на разрывы страниц/структуру, эмодзи в Times New Roman = «иероглифы»,
а скилл велел писать «что заполнить/риски» ПОСЛЕ договора → запекалось в файл. Решение
(Albery `dba1630`→`a7efe7d`):
- **MCP-тул `export_document(title, html, format=docx, font_size_pt, line_spacing)`** — агент
  сам собирает ПОЛНЫЙ HTML документа (page-break-before:always для приложений, text-align,
  text-indent:1.25cm, font-size, `<table border="0">` для реквизитов двумя колонками), новый
  модуль `docformat.py` (без зависимостей, html.parser + python-docx) конвертит механически:
  А4, поля ГОСТ (лево 3см), Times New Roman всюду, **эмодзи вырезаются на уровне конвертера**.
  Фидбек юзера по оформлению агент чинит правкой СВОЕГО HTML — без деплоев кода. Файл — в тот же
  export-механизм (`_b24_save_export` → token-ссылка mcp.m4s.ru). Лимит html 400к, только docx
  (pdf/xlsx остаются на старых маркерах [[DELIVER_*]]; промпт ФАЙЛ-ОТВЕТ говорит: официальный
  документ Word → export_document, остальное → маркеры).

  ТОЛЬКО через export_document, в html — только сам документ (мета «что заполнить/риски» —
  отдельным сообщением в чате, НЕ в файле), явная нумерация пунктов в тексте, шаблон шапки/
  реквизитов/приложений.
- **Дыра: тело кастомного скилла не доходило до модели.** Кастомные (shared) скиллы живут

## История диалогов пер-агент — каждый бот свою, не смешивать (2026-07-04)
Владелец: у каждого агента должен быть СВОЙ список диалогов, не сваливать всё в общую историю.
Было: левая панель «Агенты» в `/agent-dialogs` ставила `activeAgentId`, но НИЧЕГО по нему не
фильтровала (стоял комментарий-заглушка «на будущее»), а `/api/agent-center/dialogs` вообще не
смотрел на `agent_slug` → переписки всех ботов в одной куче. Сделано и на проде (коммит `e9ec567`):
- **Модель данных:** ходы субагента пишутся с `agent_slug='<slug>'`, ходы Универсального (основного)
  агента — с `agent_slug IS NULL` (в `b24bot._b24_log_interaction`, `agent=None` для основного).
  ⚠️ **КЛЮЧЕВОЕ (исправлено `46d4d40`):** в Bitrix приватный чат с ботом ключуется по ПОЛЬЗОВАТЕЛЮ
  (`dialog_id = user id`), поэтому ОДИН и тот же `dialog_id` переиспользуется КАЖДЫМ ботом, кому
  этот человек пишет (реально на проде: у `dialog_id=22` — 130 ходов main + 3 agent-sklad + 1
  alberi-test). Значит «диалог» = пара **(agent_slug, dialog_id)**, а не просто dialog_id.
- **Бэкенд list** `/api/agent-center/dialogs?agent=<slug|main|all>`: `main` → `agent_slug IS NULL`,
  `<slug>` → `agent_slug=<slug>`, `all`/пусто → без фильтра. Группировка/`DISTINCT ON` по
  **(agent_slug, dialog_id)**, ОБА CTE (`last`+`agg`) и поиск скоупятся выбранным ботом → превью,
  turns, errors — только этого бота. Каждая строка отдаёт `agent_slug` (NULL → `'main'`). Карточка
  основного в `/agents` тоже считает только свои ходы (`agent_slug IS NULL`).
- **Бэкенд thread** `/api/agent-center/dialog-messages?dialog_id=..&agent=..`: лента тоже фильтруется
  по `agent_slug` (main → IS NULL). БЕЗ этого лента одного бота подтягивала ходы других ботов того
  же пользователя (баг в скрине владельца: под «Агент-юрист» показывалась переписка Универсального).
- **Фронт:** `fetchAgentDialogs` шлёт выбранного агента; `DialogsView` перезагружает список при
  смене бота и сбрасывает открытый диалог + старый список (эффект на `activeAgentId`). `Chat.id` =
  композит **`${agentId}:${dialogId}`** (уникален на бот+юзер, т.к. dialog_id общий), `Chat.dialogId`
  = сырой id; лента тянется `fetchDialogTurns(dialogId, agentId)`; шапка показывает сырой dialog_id.
- **Проверено на проде (`46d4d40`):** `agent=alberi-test` → `[(alberi-test,22,1)]` (1 диалог, 1 ход,
  НЕ 130), `agent=agent-sklad` → 3 хода, `agent=main` → 4 диалога только со своими счётчиками,
  `agent=<новый/несуществующий>` → **`[]` (0 диалогов)** ← ровно требование владельца «создал агента
  = 0 переписок». Лента `dlg22`: alberi-test 1 ход / agent-sklad 3 / main 130 — без протечки.
  `agent=all` → каждая пара (бот,юзер) отдельной строкой, ничего не смешано.
- ⚠️ **Грабля деплоя (2026-07-04) — НЕ сервер, а Git Bash:** `_deploy_helper.py prod --put` из
  Git Bash падал `put → ENOENT`. Причина найдена по verbose-логу `sftp-server` на 186: сервер

## Никаких зависаний: отмена хода, fail-fast, дисциплина пакетов (2026-07-06, Albery `a0b6f51`)
Владелец: «новая сессия должна останавливать процессы; контекст не должен разрастаться; агент
не должен виснуть на невозможном (Докучаева у Артура) и тянуть до таймаута молча (юрист)».
Диагноз по журналу/интеракциям (важно — НЕ гипотезы):
- **«Таймауты» юриста (327/334/339/341/362, ~590с)** = НЕ нехватка инструментов, а **Broken pipe
  до Codex**: внутри hermes 3 ретрая (~290с) + полный повтор хода b24bot (~290с) → юзер ждал
  10 минут ради «Что-то временно сбоит». Тот же запрос при здоровом Codex прошёл за 139с (358).
- **Артур (355, 600с)**: подтвердил постановку 6 задач, среди исполнителей несуществующая
  «Анастасия Докучаева» → агент воевал с «не найден» до потолка 600с вместо «поставил 5,
  эту не знаю — уточните».
Решение (4 слоя):
1. **«Новая сессия» убивает живой ход** (`_LIVE_TURNS` реестр по scope + `_b24_cancel_live_turns`;
   `_hermes_run_guarded` переведён на Popen с communicate(timeout)): kill hermes-subprocess →
   воркер получает `cancelled` → молча убирает статус-месседж и НЕ постит ответ; reset отвечает
   «⏹ Остановил текущую работу и начал новую сессию…». Смоук на проде: cancelled=1,
   result=(None,'cancelled'), реестр чист.
2. **Fail-fast при сбоях LLM**: `agent.api_max_retries: 3→2` в конфиге hermes (186) — честная
   ошибка юзеру теперь за ~6-7 мин worst-case вместо 10 (а чаще attempt 2 b24bot просто успевает).
3. **Ошибки «не найден» = команда к действию** (context_server, все 4 места резолва людей):
   «НЕ повторяй вызов с этим именем, НЕ подбирай замену сам — сделай остальные, спроси у
   пользователя, кто это». Ошибка тула — самый сильный рычаг против лупов.
4. **Маршрутная карта += «Пакет действий»** (universal, все агенты): по одной; не получилось —
   пропусти; неудачный вызов с теми же аргументами не повторять; «не найден человек» → пример
   ответа с уточнением; «никогда не тяни до таймаута молча».
Плюс: **кап истории в промпте** (вопрос 500 / ответ 1500 символов) — договоры в истории больше
не раздувают каждый следующий ход. «Сессия не разрастается» теперь по построению: свежая сессия
на ход (`e3a9d9b`) + rolling summary + кап истории; компрессию hermes НЕ включали намеренно.
128 тестов, deploy_smoke OK, журнал чист. Остаточная правда: Broken pipe сам по себе — сеть/
провайдер, полностью не убирается; убрано ДОЛГОЕ МОЛЧАНИЕ вокруг него.

## Модель агентов (согласовано владельцем 2026-07-03)
Одна модель «Агент», один редактор у ВСЕХ. Отличаются боты только содержимым трёх наборов
(MCP-инструменты · инструкции · скиллы) + промпт + кто пользуется. Роли: **Универсальный**
(основной, самый широкий набор, для всех) · **Отделовой** (узкий срез под задачу, участники
отдела) · **Разработчик** (admin-инструменты, 1–2 человека). Общий контекст = все боты читают
ОДНУ базу знаний/одни MCP-инструменты; у Универсального просто ничего не урезано, отделовому
режем набор. Топология на старте: **Универсальный + Разработчик**, отделовые — точечно по
необходимости (не плодить заранее).

### Универсальный (основной) агент как первоклассный настраиваемый Агент (этап 2, LIVE 2026-07-03)
Основной бот теперь — обычная строка `agents` (`slug='main'`, уровень `ops`, БЕЗ admin), тот же
редактор и энфорсмент, что у субагентов, просто самый широкий. `agent_center.ensure_main_agent()`
идемпотентно создаёт строку + коннектор `/mcp-agent/main` при импорте (best-effort, не фатально).
⚠️ `bitrix_bot_id` у строки `main` = NULL НАМЕРЕННО, чтобы `agent_for_bot_id` не резолвил основной
бот в субагента (основной ход сохраняет общий промпт ассистента, не «специализированный»).
Маршрутизация за флагом: `universal_main_connector()` отдаёт `agent-main` только если
`UNIVERSAL_MAIN_AGENT=1` (в `/var/www/albery/.env`) И строка+коннектор готовы, иначе `None` →
классические тир-коннекторы (фолбэк, ничего не ломается). В `b24bot.hermes_brain_answer` при
готовности основной ход идёт на ОДИН набор `agent-main` (ops, без admin) для всех допущенных,
tier форсится в `ops` (у admin-юзеров тоже — сильные права ушли в отдельного «Агента-разработчика»,
которого владелец создаёт сам). Доступ (кто вообще общается) остаётся на `agent_access` (`none` =
молчим). Основного агента нельзя удалить; он исключён из списка субагентов; PATCH `main`
сохраняет имя (+реймами Bitrix-бота)/промпт/активность. UI: read-only витрины основного редактора
скрыты, вместо них живой `AgentCapabilityPanel slug='main'`.
**Rollout выполнен безопасно:** деплой «спящим» (флаг off) → изоляционный смоук коннектора → флип
`UNIVERSAL_MAIN_AGENT=1` + рестарт → реальный ход основного бота (admin-юзер → ops) ответил за ~12с
→ `deploy_smoke` OK. Откат: убрать флаг из `.env` + рестарт (мгновенно вернёт тир-коннекторы).
