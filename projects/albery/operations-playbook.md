## 1. Groq: для чего на самом деле
Groq используется НЕ только для аудио. Две роли:
1. **STT** — расшифровка голосовых (`whisper-large-v3-turbo`). Работает, не трогать.
2. **Auxiliary (aux)** — служебные мелкие LLM-вызовы: заголовки, подбор скиллов, web-extract, триаж и
   раньше **сжатие контекста**. Все aux висели на Groq free (`llama-3.3-70b-versatile`, ~12k токенов/мин).

**Сжатие контекста** = когда контекст разрастается, шлюз отдельным LLM-вызовом ужимает старую часть в
саммари, чтобы влезть в окно модели. На больших контекстах (генерация недельного отчёта — десятки тысяч
токенов) бесплатный Groq захлёбывался → `payment/credit error` → сжатие ломалось и **выкидывало правила
из памяти модели**. **2026-06-14 сжатие ОТКЛЮЧЕНО** (`compression.enabled: false`): окна модели хватает,
есть страховки `hygiene_hard_message_limit: 50` + `session_reset`. STT и мелкие aux на Groq остались.
Groq-ключ Albery — в `/root/.hermes/secure/hermes-gateway.env` (`GROQ_API_KEY`); проверять SDK-путём
(`OpenAI(base_url=.../groq...).models.list()`), не curl.

## 2. Недельный отчёт собственнику — КАК он генерится (ключевое!)
⚠️ **ДВА разных источника промпта — главная ловушка этой сессии.** Если отчёт «не той структуры» —
СНАЧАЛА пойми, каким путём он сделан (у строки `owner_weekly_reports`: `prompt_id` пуст → агент;
заполнен → UI-пайплайн):
1. **Агент** (просишь Hermes в Telegram / cron) → читает инструкцию `ai_instruction_folders /
   «Еженедельный отчет по компании»` через `start_here`, пишет `report_text` сам, сохраняет MCP
   `save_owner_weekly_report` (в БД `prompt_id=NULL`, `source=mcp_save_owner_report`).
2. **UI-кнопка** (Albery «Сводная аналитика») → активный промпт в `ai_prompts` (категория
   `owner_weekly_report`, строка `b941c658`); app берёт `report_text` из ответа LLM напрямую
   (функция `_build_owner_weekly_report_text` в app.py — лишь fallback на пустой `report_text`).
   `get_report_contract owner_weekly` отдаёт пусто — это НЕ значит «контракта нет».

**ОБА источника обязаны держать одну v3-структуру.** Правишь один — синхронизируй второй. Бэкапы:
`/root/.hermes/secure/weekly_instr.bak.*` (инструкция), `/root/.hermes/secure/owner_weekly_prompt.bak.*`
(UI-промпт). В `ai_prompts` НЕТ колонки `updated_at` (UPDATE без неё).

### Эталон структуры (v3) — что хотим всегда
Заголовок + период (рабочая неделя **пн–чт; пятница выходной; контроль в четверг** — отсутствие
созвона в пт НЕ нарушение). 10 секций: (1) вердикт + «что принять / что нельзя», (2) динамика+цифры,
(3) ритм встреч — ТАБЛИЦА, (4) соответствие регламентам — ТАБЛИЦА, (5) поведенческие — ТАБЛИЦА,
(6) **оценка руководителей Наталья/Артур — ТАБЛИЦАМИ по каждому** (`Задача (план+вес) | Факт (таблица↔
созвон) | Вердикт ✅/⚠️/🚫/❌ | Артефакт`) — это ядро, по строке на каждую задачу, (7) несостыковки
созвон↔таблица, (8) план следующей недели, (9) решения собственника (НЕ ставить Александра
ответственным), (10) итог. Оговорки обязательны: пятница-выходной; **аномалия Битрикса** (нет новых
задач → подсветить вероятную истёкшую подписку Bitrix Marketplace, не выдавать старые цифры за
актуальные); ФИО по транскрибации (имя аккаунта «Координатор» — не нарушение).

### Как проверить готовый отчёт (без догадок)
Вытащить из БД `owner_weekly_reports` (ORDER BY `generated_at` DESC, НЕ по `id` — это UUID): смотреть
`report_text`, наличие `| Вердикт` (таблицы раздела 6), оговорки. `is_current=true` — текущая версия;
старые версии остаются (versioned).

## 3. Почему отчёт выходил «то густо, то пусто» (корневой разбор)
Промпты были правильные (хорошая v3 16:47 — доказательство). Ломала **связка Groq-сжатие + большой
контекст**: при генерации агент тянет горы данных → сжатие → Groq падает → правила таблиц теряются →
текст вместо таблиц. Когда Groq жив — таблицы есть, когда нет — нет. **Лечение (сделано): отключить
сжатие** (раздел 1) — инструкция всегда целиком в контексте.

## 4. Рантайм-здоровье: как диагностировать «тупит / делает фигню»
**Автоматизировано (2026-06-14):** на 186 стоит ежечасный cron `self-check` (`4e496f64fade`,
`/root/.hermes/scripts/hermes_selfcheck.py`, no-agent) — сам сканит журнал и шлёт Александру в Telegram
алерт, **только если** найден сбой (codex-лимит / Groq unhealthy / падение сжатия / медиа-дроп); тихо,
когда чисто. То есть про лимит codex и падение Groq узнаёшь сразу, а не постфактум. Ниже — что значат
сигнатуры (для ручного разбора `journalctl -u hermes-gateway`):
- `Auxiliary: marking local/custom unhealthy (payment / credit error)` → Groq aux упал (лимит/ключ).
- `HTTP 429 usage_limit_reached` (`openai-codex`) → **лимит ChatGPT исчерпан** (1 аккаунт, без фолбэка;
  сброс ~раз в N часов). Сообщения провайдера теперь внятные (патч `hermes_provider_error_patch.py`:
  лимит / нет аккаунта / rate-limit — `_gateway_provider_error_reply`).
- Доставка файлов (PDF и т.п.): должны лежать в allowlist `gateway.media_delivery_allow_dirs`
  (для Albery добавлен `/root/.hermes/media_cache` — туда падает PDF отчёта). Иначе молчаливый дроп.

## 5. Как правильно работать (золотые правила сессии)
1. **Диагностируй ПРЕЖДЕ чем менять.** «Отчёт плохой» → найди, каким путём и каким промптом он сделан
   (раздел 2), прочитай сам промпт/инструкцию и сам отчёт из БД. Не правь вслепую.
2. **Помни про ДВА источника** недельного отчёта. Не путай агентскую инструкцию и UI-промпт `ai_prompts`.
3. **Проверяй бэкенды** (codex-лимит, Groq aux) — часто «фигня» это рантайм, а не промпт.
4. **Сначала бэкап, потом правка** (config/инструкция/промпт), рестарт только `hermes-gateway`, не трогать
   `albery`/БД. Доступ к 186 — джамп через Vault.
5. **Один аккаунт codex = хрупко.** Тяжёлые/частые прогоны жгут 5-часовой лимит. Не дёргать генерацию
   многократно. На будущее — второй Codex-аккаунт в пул (`fill_first` уже настроен).
