# Albery — агентская система: как работать, чтобы НИЧЕГО не сломать

> ⚠️ **ЧИТАТЬ ПЕРЕД ЛЮБОЙ РАБОТОЙ с агентами Albery.** Это главный свод правил.
> Аудит надёжности пройден 2026-07-06: журнал чистый, изоляция проверена вживую,
> e2e-смоук пройден. Ломается система только руками — поэтому правила ниже жёсткие.
> Детальная история строек — `projects/albery/agent-center.md`.

## 1. Устройство на пальцах

**Агент = строка в таблице `agents` + Bitrix-бот + личный MCP-коннектор.** Всё.

```
Сотрудник в Bitrix → бот (BOT_ID) → b24bot.agent_for_bot_id() → hermes -z -t agent-<slug>
                                                                   ↓
                                    /mcp-agent/<slug>/<mcp_token>  ← ЕДИНСТВЕННАЯ дверь агента
```

**Агенты на проде (2026-07-06):**
| slug | имя | tier | команда | инструменты |
|------|-----|------|---------|-------------|
| `main` | Агент Албери | ops | все с `agent_access`≠none | 76 (кастом) |
| `agent-sklad` | Агент-юрист | faq | id 16, 22 | 27 (кастом) |
| `agent-razrabotchik` | Агент-разработчик | **developer** | id 14 Палей, 16 Никитенко | **87 = ВСЁ** |

**Агент-разработчик — главный.** `tier='developer'` + `tools_customized=false` ⇒ его набор =
`set(TOOLS)` целиком (все admin-инструменты + 6 инструментов управления агентами
create/update/delete_agent, set_agent_tools, set_agent_team, list_agents + самообучение +
автоматизации). **Новые инструменты реестра прилетают ему АВТОМАТИЧЕСКИ.**
⚠️ Поэтому НИКОГДА не кастомизируй его тулзы в панели «Возможности» — первое же сохранение
поставит `tools_customized=true`, и автообновление набора отключится. Если случайно
кастомизировали — вернуть: `UPDATE agents SET tools_customized=false WHERE slug='agent-razrabotchik'`.
Его команда — строго 1–2 доверенных человека (через него можно управлять всей системой).

## 2. Пять слоёв изоляции (все проверены вживую 2026-07-05/06)

1. **Инструменты** — коннектор `/mcp-agent/<slug>/<token>` отдаёт в `tools/list` РОВНО набор
   агента; выключенный инструмент невидим и невызываем (движок, не промпт).
2. **Инструкции** — scoped `start_here`: universal ∪ подключённые в манифесте
   `agent_knowledge/agents/<slug>.yaml`. Неподключённая optional физически недоступна.
3. **Сессии** — каждый ход `hermes -z` one-shot: без чужого контекста, без памяти между агентами.
4. **Данные self-тулов** — `upsert/list/delete_my_instruction`, `schedule/list/delete_my_automation`
   скоупятся slug'ом ИЗ URL коннектора — чужие записи недостижимы.
5. **Диалоги и мониторинг** — «диалог» = пара `(agent_slug, dialog_id)`: истории ботов не
   смешиваются (в Bitrix dialog_id приватного чата = id юзера и ПЕРЕиспользуется всеми
   ботами!). Мониторинг тоже пер-агентный (`?agent=`): карточки/график/лента — только
   выбранного бота; «Здоровье систем» намеренно общее (одна инфраструктура на всех).

**Мягкий край (единственный, помнить):** скиллы Hermes из `/root/.hermes/skills/` gateway
рекламирует модели глобально — селекция скиллов у агентов = промпт-энфорсмент. Жёсткими
остаются инструменты и инструкции; большинство скиллов без своих MCP-тулов инертны.

## 3. Создание / удаление агентов — ТОЛЬКО два пути

✅ **UI** (Центр Агента → Агенты → «Создать агента») или ✅ **чат с разработчиком**
(инструменты `create_agent` / `delete_agent` — есть только у него).

Оба пути делают ВСЁ атомарно: Bitrix-бот (`imbot.register` через то же локальное приложение),
строка `agents` + `agent_members`, коннектор в `/root/.hermes/config.yaml` (с бэкапом +
yaml-валидацией; рестарт Hermes НЕ нужен). Удаление — зеркально: unregister бота, чистка
коннектора, каскад БД **включая автоматизации агента** (иначе сирота молотила бы вечно).

❌ **НИКОГДА руками:** INSERT/DELETE в `agents` мимо приложения; правка секции `mcp_servers`
в `config.yaml` вручную; удаление бота напрямую в Bitrix (событие ONIMBOTDELETE от чужого бота
раньше стирало state ВСЕХ ботов — защита стоит, но не искушать).

**Проверка после создания (30 секунд):** строка в `agents` с `bitrix_bot_id` не NULL →
`grep agent-<slug> /root/.hermes/config.yaml` = 1 → `tools/list` коннектора отдаёт ожидаемое
число. Кэш агентов в сервисе 60с — изменения в БД видны коннектору не мгновенно.

## 4. Изменение возможностей агентов

- **Инструменты** — панель «Возможности» в редакторе (или `set_agent_tools` у разработчика).
  Admin-инструменты (бейдж `admin`) включать только агентам с узкой доверенной командой:
  caller-id до MCP-тула не доходит, безопасность = ограниченная команда.
- **Инструкции** — источник правды `agent_knowledge/` в git-репо Albery; редактировать контент
  через UI «Инструкции для ИИ» (БД→git зеркалится автоматом + push). ⚠️ Прямые правки файлов
  на GitHub перезапишутся следующей правкой из UI. Scope: `universal` = всем, `optional` =
  по манифесту. Личные — файлы `agents/<slug>/learned/*.md` с атрибуцией.
- **Команда** — редактор агента или `set_agent_team`. У `main` команда = `agent_access`
  (⚠️ «забрать доступ» = строка `none`, НЕ удаление строки; без явного гранта бот молчит).
- **Автоматизации** — вкладка «Автоматизации» или self-тул из чата. Капы: self ≤24 зап/сутки
  и ≤10 штук, владелец ≤96/сутки. Системные строки (`kind='system'`) — read-only витрина
  Hermes-кронов, приложение их НЕ исполняет: управлять ими только в Hermes на 186.

## 5. Изменение КОДА — конвейер (не отступать ни на шаг)

1. Ветка в локальном клоне `C:\Users\hotiz\albery-audit` (клон «Евгений. Разработка» МЁРТВ).
2. Сборка/тесты ЛОКАЛЬНО: `tsc` + `vite build` + `pytest` (~125 pass). **На боксе (2 ГБ RAM)
   НИЧЕГО тяжёлого не собирать** — `engineering/server-preflight.md` обязателен.
3. Merge в main (ff) → push (CI: миграции Postgres + pytest + фронт).
4. На 186: `git pull` → `scripts/ensure_postgres.py` (если миграции) → `systemctl restart albery`
   (ТОЛЬКО если менялся бэкенд) → **ОБЯЗАТЕЛЬНО `scripts/deploy_smoke.py`**.
5. `dist/` в git не живёт: tar локальной сборки → SFTP `_deploy_helper.connect("prod")`
   (из PowerShell или `MSYS_NO_PATHCONV=1` — Git Bash корёжит `/tmp/...` в виндовый путь).
6. **Миграции применяются ТОЛЬКО из реестра** `ensure_postgres.py` (файл в папке сам НЕ
   применится); номер может быть занят — прод-агент сам добавляет миграции, сначала
   `ls database/migrations | tail`. Прод коммитит и пушит сам (deploy key) — перед работой
   всегда `git pull`, дерево держать чистым.
7. ❌ **Не добавлять код в `app.py`** (заморожен, Ш5) — новые модули паттерном b24bot/agent_center
   (регистрация роутов при импорте, в app.py только `import <module>`). В agent_center импорты
   из b24bot — только ленивые (циркулярка); в скриптах ВСЕГДА `import app` первым.
8. Bitrix-вебхук: до HTTP-ответа НИКАКИХ внешних вызовов (ACK сразу, всё сетевое — в worker),
   иначе Bitrix считает бота мёртвым (инцидент 499, 2026-07-04).

## 6. Ресурсы и стабильность (бокс 2 ГБ!)

- Полосы исполнения РАЗДЕЛЕНЫ: сотрудники ≤3 слота hermes, автоматизации — своя очередь
  ≤1 воркер (`AGENT_AUTOMATION_CONCURRENCY=1` — **НЕ поднимать**, worst case уже 4×hermes
  ≈ 1 ГБ). Никто никого не блокирует, запуски не дропаются (очередь + 1 ретрай через 120с).
- Зависший `running` (рестарт посреди хода) самолечится → `interrupted`.
- Kill-switches: `AGENT_AUTOMATIONS=0` (весь планировщик), `AGENT_HEALTH_WATCHDOG=0`,
  убрать `UNIVERSAL_MAIN_AGENT=1` из `.env` + рестарт = мгновенный откат main на тир-коннекторы.
- Здоровье смотреть: `/agent-monitoring` (7 систем + лента), вотчдог сам шлёт TG в
  Albery_Уведомления при проблемах и «✅» при восстановлении.

## 7. Если что-то сломалось (симптом → действие)

- **Бот молчит** → это ДОСТУП, не поломка: у `main` юзер должен иметь строку `agent_access`
  ≠ none; у субагента — быть в `agent_members` (пустой список = все допущенные).
- **Бот «выключен»** → тумблер is_active в редакторе.
- **Пропал application_token / бот умер после удаления другого бота** → самовосстановление:
  пингнуть бота вебхуком `im.message.add {DIALOG_ID: <bot_user_id>}` — state пере-принимается.
- **Новый агент не видит инструмент** → подождать 60с (кэш) → проверить `tools/list` коннектора.
- **Автоматизация «прервана»** → норма после рестарта, run-now разблокирован.
- **«SFTP сломан» ENOENT** → это Git Bash MSYS, не сервер: `MSYS_NO_PATHCONV=1` или PowerShell.
- **Silent-деградации мозга** → `journalctl -u hermes-gateway`, смотреть token_invalidated /
  SOUL blocked / compression fail (`logs/mistakes.md` Hermes-мозга).

## 8. Открытые мягкие места (честный список, 2026-07-06)

1. Скиллы Hermes глобальны для модели (промпт-край, п.2). Движковое скрытие = правка gateway.
2. Устаревший **исходящий вебхук задач** в портале шлёт на старый секрет → 403×6/день (шум,
   рабочий вебхук параллельно отвечает 200). Удалить руками: Bitrix24 → Разработчикам →
   Другое → Исходящие вебхуки → обработчик `/bitrix/events/tasks/12f8f0c6…`. REST его не видит.
3. `red-teaming/godmode` выпилен из живых скиллов в `/root/.hermes/skills_disabled/red-teaming`
   (2026-07-06). ⚠️ После `hermes update` бандл может вернуть его — проверять и снова убирать.
4. Инжект подключённых скиллов у main в живых ходах не сделан (в автоматизациях — у всех есть).
5. Точные токены хода — постфактум-матчинг state.db (можно писать из b24bot напрямую).

## 9. Правило для будущих сессий ИИ

Любая работа по агентам Albery: (1) прочитать этот файл + `agent-center.md`; (2) перед
изменениями — `git pull` на боксе и локально, дерево чистое; (3) после изменений —
`deploy_smoke.py` + смоук именно изменённой механики (создал агента — проверь tools/list;
поменял доступ — проверь отказ/допуск); (4) **обновить `agent-center.md` + этот файл в конце
сессии** (правило владельца) + строка в `logs/changelog.md` мозга.
