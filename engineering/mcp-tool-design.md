# Проектирование MCP-инструментов: как сделать ИИ супер-эффективным

Дистиллят марафона «Простые поставки» 2026-07-09…11 (≈15 инструментов, 4 конвейера, всё проверено
живым агентом на реальных данных). Применимо к любому проекту с агентом (Albery, будущие).

## Главный принцип

**Под каждую боль владельца делай MCP-инструмент, а не «умный промпт».** Разделение труда:
- **Сервер (инструмент)** — вся механика, детерминированно: валидация обязательных полей,
  вычисления (дедлайны из «в течение 21 дня», суммы прописью), привязки (организация по ИНН через
  DaData), файловые операции (клонирование, превью, конвертация), запреты (cp1251-санитизация).
- **Агент (LLM)** — только то, что умеет уникально: прочитать документ (текст/изображения),
  извлечь данные, принять решение, спросить владельца.

Симптом, что нужен инструмент: агент делает 5+ вызовов там, где владелец ждёт один шаг; агент
«дебажит систему»; агент делает одно и то же неточно раз за разом.

## Правила проектирования инструмента

1. **Один вызов = одна бизнес-операция целиком.** «Занести договор» = организации + карточки +
   файл + пометка документа + снапшот — внутри одного save, а не 5 инструментов по очереди.
2. **Ответ = самодостаточная проверка.** Верни агенту всё для сверки одним взглядом (itemsSaved,
   totalAmount, sumWarnings, linkedOrganizations с isPlaceholder, deadlineSaved) — тогда не нужны
   контрольные чтения, а расхождение чинится одним повторным вызовом. Пустой обязательный результат
   — warning прямо в ответе.
3. **Валидируй на входе и называй пустоты по-русски.** «ПУСТЫЕ ОБЯЗАТЕЛЬНЫЕ ЯЧЕЙКИ: дата договора
   (contractDate)…» — агент дозаполняет и повторяет; escape-hatch (allowPartial) только с явным
   докладом владельцу.
4. **Легаси-странности объявляй в ответе** (schemaNote «кол-во хранится в поле price — это НЕ
   баг»). Иначе агент примет норму за поломку и уйдёт «чинить систему» (реальный кейс: тройной
   пересохран + SSH-дебаг прода).
5. **Мультимодальность — точечно.** Страницы-изображения по запросу (pages=[1,7,13]), каждая
   страница затратна; в ответе — предупреждение «не видишь изображения → остановись и скажи»
   (антивыдумывание).
6. **Жизненный цикл данных — часть контракта инструмента.** Знай, кто и когда удаляет/переносит
   сущность (входящий документ израсходуется при создании контракта); не оставляй призраков;
   документируй цикл прямо в hint ответа.
7. **hint/nextSteps в каждом ответе** — следующий шаг конвейера подсказывает сам инструмент, а не
   память агента.
8. **Запреты — в промпте, жёстко и с объяснением**: не лезть в сервер/код/БД, не выдумывать, не
   дублировать (проверка get_contracts перед созданием).

## Как проверять (обязательный порядок)

1. Бэкенд-e2e на посеянных данных с полной очисткой (реальные данные не трогать).
2. **Живой агент на реальных данных** — он находит то, что e2e не видит: молча потерянные поля,
   выдумывание при отрезанном vision, дубли от параллельной работы владельца. Для Hermes:
   `hermes -z` БЕЗ `-t`-ограничения (иначе режется vision_analyze и агент слепнет молча);
   рискованные сценарии — dry-run («только анализ, ничего не создавай»).
3. Сверка результата с первоисточником своими глазами (страницы документов — через view + Read).

## Инфраструктурные грабли (Простые поставки, но общие по духу)

- Код между `@app.post(...)` и `def` роняет приложение — **import-check до рестарта** в деплое
  (сервис остаётся на старом коде при падении импорта).
- MCP-клиент гейтвея умирает навсегда после ~30с недоступности сервера — **рестарт гейтвея после
  каждого деплоя бэкенда** (+ новые тулы всё равно требуют рестарта для discovery).
- Легаси-потребители (Business Pack/Firebird WIN1251) не едят юникод — санитизация на выходе.
- Живая схема БД может не совпадать с CREATE TABLE в коде — проверяй реальные колонки.
- Тестовые агентские прогоны жгут квоту ChatGPT — держи задачи узкими, тесты точечными.
