# Building the Hermes agent team — grounded in proven patterns

Как строить «команду агентов» Hermes так, чтобы она была надёжной, а не «плыла» (см. инцидент
04.06.2026, разбор в памяти `hermes-multi-agent-design`). Это НЕ теория — принципы вытащены из двух
первоисточников и адаптированы под наш стек (Hermes Agent, `openai-codex`, Telegram, MCP Albery, cron,
`agent-knowledge`):

- **Anthropic, «Building Effective Agents»** — anthropic.com/engineering/building-effective-agents
  (официальный гайд: паттерны оркестрации; когда агент, а когда workflow).
- **Microsoft, «AI Agent Orchestration Patterns»** — Azure Architecture Center (офиц., обновлён 05.2026):
  sequential / concurrent / group chat / handoff / magentic-supervisor; single-responsibility; антипаттерны.
- **12-Factor Agents** — github.com/humanlayer/12-factor-agents (community-стандарт надёжности; не вендор).

Все три источника **сходятся** на одних принципах — ниже они отмечены [A]=Anthropic, [M]=Microsoft, [12].

## 0. Главный тезис (почему агенты «плывут»)

Из 12-factor: надёжный агент — это **в основном детерминированный софт, в который LLM-шаги вставлены
точечно**, а НЕ открытый цикл «LLM сам решает → выполняет → повторяет». Стандартный открытый цикл не
дотягивает до production-качества (80%+). Из Anthropic: **начинай с простого, усложняй только когда
проще не справляется**; многие задачи — это не «агент», а **workflow** (фиксированный путь с LLM-шагами).

Вывод для нас: наши cron-процессы (zoom→tasks, owner-daily) — это **workflows** (детерминированный
скрипт + точечный LLM-шаг по строгому промпту), и их нельзя превращать в открытый агентский цикл.
Открытый агент уместен только там, где шаги нельзя предсказать.

## 1. Какой паттерн выбирать (Anthropic)

| Паттерн | Что это | Где у нас |
|---|---|---|
| **Routing** | классификатор направляет запрос в профильный обработчик | **Главный** = роутер: общее/напоминания держит, Albery → отдаёт Темуру |
| **Orchestrator–workers** | центральный LLM дробит задачу, делегирует воркерам, собирает итог | Главный (оркестратор) ↔ Темур (воркер по Albery); подзадачи определяются по контексту |
| **Prompt chaining** | задача = фиксированная цепочка шагов с проверками между ними | owner-daily: читать инструкции → собрать данные → сформировать отчёт → сохранить |
| **Autonomous agent** | открытый цикл с инструментами, чекпойнты к человеку | редко; только для непредсказуемых задач, и всегда с human-checkpoint |

**Правило сложности:** сначала один промпт/один workflow. Отдельного агента заводим, только когда
проще-решение перестало справляться, а не «потому что круто иметь команду».

## 2. 12 факторов, применённые к Hermes

1. **NL → tool calls.** Намерение пользователя → структурный вызов инструмента (MCP/cron), не вольный текст.
2. **Own your prompts.** Промпты cron — в git (`scripts/hermes_*_prompt.txt`), не в дефолтах фреймворка.
3. **Own your context window.** Активно курировать контекст: компрессия (Groq) + `session_reset` +
   `telegram_context_guard`. Раздувание контекста = корень падений (и расхода лимита Codex).
4. **Tools = structured outputs.** MCP-инструменты Albery с чёткой схемой; не «магия».
5. **Unify execution & business state.** Состояние «обработано» писать в БД проекта (Albery
   `bitrix_dispatch`/`ai_report`), а не только в голове агента.
6. **Launch/Pause/Resume.** systemd + `hermes cron pause/resume`; рестарт не должен терять прогресс
   (батчить правки, рестартить один раз).
7. **Контакт с человеком — через tool call.** Подтверждение «ставь» перед записью в Bitrix — это
   human-in-the-loop как штатный шаг, а не исключение.
8. **Own your control flow.** Явная логика (скрипт-watchdog решает, звать ли LLM), а не открытый цикл.
9. **Compact errors into context.** Ошибки сворачивать коротко (наш `cron.wrap_response=false`,
   тихий режим), чтобы агент учился, а не тонул в трейсбеках.
10. **Small, focused agents.** Узкая зона у каждого агента. Один домен — один агент.
11. **Trigger from anywhere.** Точка входа — Telegram (личка/группа), cron, MCP; единый агент за ними.
12. **Stateless reducer.** Сессия = чистая трансформация: `session_reset` (idle 30м / 04:00), чтобы
    агент не тащил недельную историю.

## 3. Правила НАШЕЙ команды (решения, обязательные при сборке)

Грунтуются на факторах 3, 7, 10 и паттернах routing/orchestrator-workers, плюс уроки 04.06:

1. **Один сквозной workflow = один агент.** Albery Zoom→Bitrix (обе фазы), owner-отчёты, Albery-cron,
   Albery MCP — целиком у Темура. Никогда не разрывать конвейер между агентами (это и сломалось 04.06).
2. **Делить только настоящие домены** (свой MCP/данные/ответственность), не generic-способности
   [M-антипаттерн: «добавлять агентов без реальной специализации»]. Напоминания/поиск/общий чат — у
   Главного. (Никита отдельным агентом не нужен.)
3. **Целевой состав: Главный (оркестратор + общее + напоминания) + Темур (весь Albery).**
   Третьего — только со своим доменом И своим бюджетом мозга.
4. **Каждому агенту с реальной LLM-нагрузкой — свой аккаунт/ключ** [M: «общий MaaS-эндпоинт → rate-limit
   при параллельных агентах»; модель — под сложность задачи]. Иначе делят один 5ч-лимит Codex.
5. **Рабочий aux до масштабирования.** Компрессия/саммари на дешёвой быстрой модели (Groq,
   `llama-3.3-70b-versatile`), иначе петля рестартов. (Включено 04.06.)
6. **Telegram: один читатель группы — оркестратор.** `observe_unmentioned_group_messages: true`
   только у Главного; специалисты — строго по @упоминанию или по делегированию. В SOUL специалиста
   нет «отвечай на любой привет».
7. **Mention-паттерны не пересекаются**; при пересечении решает только оркестратор.
8. **Правки конфигов батчить, рестартить один раз.** Снимок (`tar`) перед серверными изменениями.
9. **ACI важнее фреймворка.** Инструменты/скиллы — с чёткими описаниями, примерами, ограничениями
   параметров (poka-yoke), как HCI. Жёсткие запреты в AI-инструкции, а не уговоры (пример Albery:
   «НЕ вызывай create_bitrix_task / НЕ читай get_org_structure»).

## 4. Каналы доставки ≠ агенты (решение по уведомлениям, 04.06.2026)

Прежде чем заводить «агента уведомлений», прогоняем по чек-листу: уведомления — это **пункт
доставки**, а не отдельный разум. Поэтому правильное решение — **отдельный канал, а не второй мозг**
[M штатно поддерживает доставку в отдельный канал; [12] factor 11 «meet users where they are»].

Принятая схема:
- **Диалоговый чат** (личка с Главным) — только живой разговор, никаких авто-доставок.
- **Чат «Уведомления»** — отдельный Telegram-чат, тот же главный бот; туда уходит **весь
  автоматический вывод**: напоминания, cron, mail-watch, дайджесты, алерты.
- Напоминание/автоматизацию ставишь Главному в диалоге → он создаёт cron с **доставкой в «Уведомления»**.
  Мозг остаётся **один** (не нарушает «пока 1 агент»), а каналы разнесены.
- Технически: главный бот добавлен в отдельный чат; `TELEGRAM_HOME_CHANNEL` / `deliver` cron-джобов
  указывают на chat_id «Уведомлений»; диалоговая личка остаётся интерактивной.
- **Реализовано 04.06.2026:** канал = группа «Уведомления», `chat_id -5120862157`; в `/root/.hermes/.env`
  `TELEGRAM_HOME_CHANNEL=-5120862157`; у всех cron-джобов с `deliver: origin` поле `origin.chat_id`
  переведено на группу (бэкапы `.env.bak.notif_*`, `cron/jobs.json.bak.notif_*`); личка `1451982360`
  оставлена чистой для диалога. Проверено: `hermes send` в группу = `sent`.
- **Интерактивность группы (04.06.2026):** в `config.yaml` для группы `require_mention: false` +
  `observe_unmentioned_group_messages: true` + `allowed_chats`/`group_allowed_chats: '-5120862157'` —
  бот отвечает на любые сообщения владельца без `@`, поэтому согласование идёт по-человечески
  («да, ставь», «перенеси», «поменяй текст»). Это безопасно, т.к. в группе **один** бот (не толпа).
  В SOUL Главного закреплено правило: напоминания/cron/автоматизации доставлять ТОЛЬКО в эту группу,
  личка — только диалог; фидбек и правки выполнять прямо в группе. Проверено владельцем: оба чата ок.

Когда понадобится отдельный **думающий** агент (например, Темур по Albery) — тогда это полноценный
агент со своим доменом и бюджетом мозга. Уведомления — не тот случай.

## 5. Чек-лист перед добавлением нового агента

- [ ] У него есть **отдельный домен** (свой MCP/данные/зона), а не generic-способность?
- [ ] Весь его сквозной workflow **помещается в нём целиком** (ничего не разрывается с другими)?
- [ ] Есть **свой бюджет мозга** (аккаунт/ключ), не отнимает лимит у других?
- [ ] Telegram-роль: читает группу **только оркестратор**; у нового — @упоминание?
- [ ] Mention-паттерны **не пересекаются** с существующими?
- [ ] Это вообще **агент**, или хватит **workflow** (cron + строгий промпт)?
- [ ] aux-компрессия работает (Groq), снимок снят, правки забатчены?

Если хоть один пункт «нет» — не заводить отдельного агента; оставить как workflow или зону оркестратора.
