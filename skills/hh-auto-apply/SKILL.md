# Skill: hh-auto-apply — поиск и автоотклики на hh.ru

Для разового ручного поиска, безопасного fallback без локального Chrome и проверки свежести/удалёнки использовать `references/manual-search-and-verification.md`.

Цикл: искать → фильтровать по профилю → писать короткий человечный отклик →
**(режим review, ТЕКУЩИЙ)** слать кандидатов владельцу в TG на одобрение /
**(режим apply)** откликаться самому через залогиненный браузер → вести журнал →
отчитываться в Telegram. Запускается по cron каждый час; владелец логинится в
браузер, когда слетает сессия.

## ТЕКУЩИЙ РЕЖИМ (apply) — автоотклики без сопроводительного
С 2026-06-24 владелец снова включил **автоотклики** по всей России на вакансии,
связанные именно с автоматизацией через ИИ, ИИ-агентами, LLM-интеграциями,
no-code/low-code, n8n/Make/Zapier и автоматизацией бизнес-процессов. Не брать чистую
ML/DS-разработку, обычный backend/Python с AI в названии, продажи, маркетинг,
контент-заводы, поддержку, 1С и управленческие роли без внедрения. Активный cron:
`hh-auto-apply`, каждые 60 минут, доставка только в Telegram-группу «Уведомления».
Конфиг `/root/.hermes/state/hh_auto_config.json`: `mode=apply`,
`attach_cover_letters=false`, `area=113`, `salary_from=100000`. Отклик делается без
сопроводительного письма; LLM всё равно может генерировать письмо как внутренний
текст для review/manual, но при apply скрипт передаёт пустой текст в форму hh.
Watcher `hh-ai-jobs-watch` поставлен на паузу, чтобы не было дублей уведомлений.

Фидбек владельца по номерам вакансий нужно использовать для подкрутки фильтра в
`hh_ai_jobs_watch.py`: расширять/сужать запросы, `EXCLUDE_RE`, `STRONG_ROLE_RE` и порог
релевантности. Профиль остаётся прежним: удалёнка, Россия, от 100к, внедрение AI/LLM
агентов и автоматизаций в бизнес, без продаж, маркетинга, 1С, поддержки, чистого ML/DS
и управленческих ролей.

Фидбек 2026-06-13: «AI-native Python Backend Developer / AI-нативный Python Backend
Инженер» (Novakid) — не полезно, потому что это выглядит как чистая backend/Python-
разработка с AI в названии; такие роли понижать, если в названии нет внедрения,
интеграции, автоматизации, n8n/Make/Zapier/workflow/no-code. «Специалист по ИИ»
(rodinka.recruitment) — топ; роли формата «специалист по ИИ», внедрение ИИ,
AI-интегратор, AI Automation Specialist поднимать выше.

## РЕЖИМ ОБУЧЕНИЯ (mode=review) — старый полуавтоматический сценарий
С 2026-06-12 автоотклики на паузе после ложных срабатываний (КСД Групп — нет ИИ
и опыт 5+; HiPo PropTech — менеджмент недвижимости с ИИ-обёрткой). Старый review-прогон
шлёт владельцу в TG список «🔎 Кандидаты на отклик» (нумерация + `id`).
Владелец отвечает в чате, Гермес (серверный агент) обязан:

1. **«N норм / откликнись на N»** → запустить отклик по одобренным id:
   ```bash
   /usr/local/lib/hermes-agent/venv/bin/python \
     /root/.hermes/agent-knowledge/skills/hh-auto-apply/scripts/hh_auto_apply.py \
     --apply-ids <id1>,<id2>
   ```
   (id брать из строки `id NNNNNN` кандидата; скрипт сам возьмёт сохранённое письмо,
   откликнется, переложит из `proposed` в `applied` и отчитается в TG).
2. **«N не норм, потому что …»** → дописать причину в фидбек-файл и убрать из
   `proposed`, чтобы не висел:
   ```bash
   python3 - <<'PY'
   import json, os
   fb = '/root/.hermes/state/hh_feedback.json'
   lg = '/root/.hermes/state/hh_applies.json'
   d = json.load(open(fb)) if os.path.exists(fb) else {"bad": [], "good": []}
   d["bad"].append("<Название (Работодатель)> — <причина владельца своими словами>")
   json.dump(d, open(fb, 'w'), ensure_ascii=False, indent=1)
   l = json.load(open(lg)); p = l.get("proposed", {}).pop("<id>", None)
   l.setdefault("skipped", {})["<id>"] = "владелец: не норм"
   json.dump(l, open(lg, 'w'), ensure_ascii=False, indent=1)
   PY
   ```
   Явную похвалу («вот такие и нужны») так же добавлять в `good`.
   Фидбек подмешивается в LLM-промпт каждого прогона и важнее общих правил.
3. **Обобщающие пожелания** («не предлагай 1С», «только удалёнка») → это правка
   `queries`/`title_exclude` в `/root/.hermes/state/hh_auto_config.json` или правил
   отбора в скрипте (через approval flow мозга).
4. Когда владелец скажет «фильтр норм, включай автоотклики» → `"mode": "apply"`
   в `/root/.hermes/state/hh_auto_config.json`.

## Профиль поиска (что нужно владельцу)
**ТОЛЬКО** внедрение ИИ и автоматизаций в бизнес, работа с бизнес-заказчиком:
ИИ-агенты/ассистенты на LLM, боты, интеграции с CRM, no-code/low-code, автоматизация
процессов и отчётности. **НЕ интересно:** ML/DS-разработка, research, чистая разработка
ПО, бигтех/гос-работодатели, продажи, маркетинг, дизайн, поддержка.
**География/деньги (решение владельца 2026-06-12):** вся Россия (`area=113`), ЗП от
100 000 ₽ — фильтр hh `salary=100000` (вакансии без указанной ЗП тоже берём; если в
тексте ЗП ниже 100к — LLM отсеивает, плюс питоновский префильтр по вилке со страницы).
Фильтр трёхступенчатый: префильтр названия/работодателя/ЗП + LLM-оценка по описанию.

## Инфраструктура (уже на сервере)
- **Браузер с сессией hh.ru:** `/opt/hh-browser-start.sh` поднимает Chrome под Xvfb
  (`:95`), CDP на `127.0.0.1:9225`, плюс VNC→noVNC для входа владельца руками.
  Профиль (куки/сессия) живёт в `/opt/hh-browser/profile` (700). Скрипт сам
  стартует браузер если не запущен и гасит, если поднимал его сам.
- **Логин — только руками владельца** (СМС/капча). Ссылку на вход даёт
  `bash /opt/hh-browser-start.sh` (noVNC URL). Пароли/коды/куки никогда не в чате,
  не в git, не в памяти.
- **API соискателя hh.ru ОТКЛЮЧЁН** (2025-12-15) и `api.hh.ru/vacancies` отдаёт
  `forbidden` с серверного IP — поэтому всё через обычные страницы `hh.ru/search`
  и DOM via CDP, как обычный залогиненный пользователь.
- **LLM-оценка и письма:** Groq `llama-3.3-70b-versatile` (`GROQ_API_KEY` в
  `hermes-gateway.env`), строгий JSON-ответ.

## Запуск
```bash
/usr/local/lib/hermes-agent/venv/bin/python \
  /root/.hermes/agent-knowledge/skills/hh-auto-apply/scripts/hh_auto_apply.py [--dry-run] [--max N] [--list]
```
- `--dry-run` — ищет, фильтрует, пишет письма, печатает их, но НЕ кликает «Откликнуться».
- `--list` — **быстрый разовый список для владельца** («найди вакансии»): поиск по всем
  кластерам запросов + префильтр (название/работодатель/ЗП из выдачи), БЕЗ LLM и откликов,
  ~30-60 секунд. Использовать его, а не ручное хождение по страницам браузер-инструментом.
- Перед запуском всегда preflight: браузеру нужно ≥250 МБ RAM. Скрипт сам проверяет и
  считает память meshcentral как доступную (start.sh останавливает его на время сессии,
  stop.sh возвращает). Автопрогоны идут с `HH_NOVNC=1` (без x11vnc/websockify, −30-40 МБ);
  для ручного логина владельца запускать `bash /opt/hh-browser-start.sh` БЕЗ переменной —
  тогда будет noVNC-ссылка. Отказ preflight теперь уходит алертом в TG (не молчит).
- Скорость (2026-07-12): ожидания страниц событийные (загрузка+селектор, а не фиксированные
  6-10с), пауза перед LLM-вызовом 6с (429 ловится retry-с-backoff). Полный прогон в разы
  быстрее; если hh начнёт недогружаться — поднять `wait`/`settle` в `Tab.goto`.

## Состояние (только на сервере, не в git)
- `/root/.hermes/state/hh_auto_config.json` — настройки: `queries` (кластеры
  запросов), `title_exclude`, `employer_exclude`, `area`, `salary_from`,
  `active_hours_msk` (окно откликов, ночью только ЛС), `max_applies_per_run`,
  `delay_between_applies_sec`, `profile` (текст для LLM), `paused`. Меняй ЕГО,
  чтобы подкрутить поиск/тон.
- `/root/.hermes/state/hh_applies.json` — журнал: `applied` (откликнулся, с текстом
  письма и датой), `manual` (требует ручного отклика — вакансии с опросом/тестом),
  `skipped` (нерелевантные/без кнопки/ЗП ниже порога). **Повторных откликов нет** —
  id вакансии в любом из трёх разделов исключает её из следующих прогонов.
- `/root/.hermes/state/hh_msgs_seen.json` — дайджесты переписок ЛС (для детекта
  новых сообщений). Первый прогон только заполняет файл, без уведомлений.

## ЛС (переписки с работодателями)
Каждый прогон сканирует `hh.ru/applicant/negotiations`: новое сообщение/приглашение →
уведомление владельцу в Telegram со ссылкой на переписку. **Отказы — молча** (правило
владельца). Статусные шумы («просмотрено») не репортятся.

## Сопроводительные по запросу работодателя
Если в описании вакансии упоминается сопроводительное письмо — агент откликается сам
(письма он пишет ко всем откликам), но дополнительно шлёт владельцу текст письма в
Telegram (блок «📩 Тут просили сопроводительное»). Если письмо прикрепить не вышло —
прямо пишет «отправь его в чате отклика».

## Письмо-отклик (анти-ИИ правила, зашиты в промпт)
2–4 коротких живых предложения, с «Здравствуйте!», зацепка за конкретную деталь
вакансии, по-человечески, без штампов («идеально подхожу», «рад возможности»,
«уникальный опыт»), без длинных тире, без подписи. Если LLM счёл нерелевантным —
вакансия уходит в `skipped`, отклик не шлётся.

## Поведение и безопасность
- Вакансии с **опросом/тестовым заданием работодателя** скрипт не проходит сам —
  кладёт в `manual` и присылает владельцу ссылку «отклкнись руками».
- **Стоп при сбоях:** 3 неудачных попытки отклика подряд → останавливается (не
  долбит сайт), делает скриншот в `/opt/hh-browser/logs/fail_<id>.png`.
- Сессия слетела → НЕ откликается, шлёт владельцу просьбу залогиниться, ставит цикл
  на паузу де-факто (login-gate в начале).
- Человеческие паузы между откликами (`delay_between_applies_sec`, 25–50 с) +
  лимит на прогон (`max_applies_per_run`, по умолчанию 8) — не выглядит как бот.
- Поставить на паузу: `paused: true` в конфиге (или `hermes cron` отключить job).

## Cron
Job `hh-auto-apply` — **каждые 60 минут** (no_agent script через обёртку
`/root/.hermes/scripts/hh_auto_apply_cron.sh`). Ночью (вне `active_hours_msk`,
по умолчанию 8–23 МСК) отклики не шлются (если браузер поднят — проверяются только ЛС;
если нет — прогон молча выходит). Чистые прогоны без новых релевантных вакансий и
сообщений молчат; есть отклики/письма/ЛС/ошибки — отчёт в Telegram.
Старый вотчер `hh-ai-business-automation-watch` (каждые 2ч, только уведомления о
вакансиях в отдельный чат) живёт отдельно и этим скиллом не управляется.

## Частые правки
- Сменить фокус поиска → правь `queries` и `title_exclude` в конфиге.
- Тон писем не нравится → правь `profile` и при необходимости анти-штамп-список в
  `llm_assess` (в скрипте).
- Селекторы hh.ru поменялись (кнопка отклика/поле письма) → правь `APPLY_JS` и
  `read_vacancy`/`search_vacancies` в скрипте; проверяй `--dry-run`.
