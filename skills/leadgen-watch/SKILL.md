# Skill: leadgen-watch — поиск проектов и клиентов (биржи + тендеры + TG-каналы)

Родной брат `hh-auto-apply`, но про **проекты**, а не вакансии: hh — канал найма,
здесь — заказчики с уже сформулированной задачей. Решение владельца 2026-07-12:
приоритетные каналы = биржи+тендеры и холодный email-аутрич; TG-каналы добавлены
как дешёвый третий канал (технология обкатана на Albery через `t.me/s/` превью).

Цикл: обход источников → дедуп по журналу → префильтр по ключам → LLM-оценка
(Groq) + черновик отклика → кандидаты владельцу в Telegram → владелец
откликается руками по черновику. **Автооткликов нет (v1 = только review)** —
на биржах нужны аккаунты и репутация; полуавто-отклики возможны позже.

Каждый источник обёрнут в try/except: падение одного не валит прогон, статусы
источников пишутся в лог (`/root/.hermes/logs/leadgen_watch.log`).

## Запуск
```bash
/usr/local/lib/hermes-agent/venv/bin/python \
  /root/.hermes/agent-knowledge/skills/leadgen-watch/scripts/leadgen_watch.py \
  [--dry-run] [--list] [--source flru|habr|workspace|telegram] [--max N]
```
- `--dry-run` — полный цикл с LLM, печать в stdout, БЕЗ Telegram и записи журнала.
- `--list` — **быстрый разовый список** («найди проекты»): обход + префильтр,
  без LLM и журнала, ~10–30 секунд. Использовать для ручных запросов владельца.
- Ночью (вне `active_hours_msk`, 8–23 МСК) кандидаты копятся со статусом
  `pending` и доезжают в TG первым утренним прогоном — ночного спама нет.

## Состояние (только на сервере, не в git)
- `/root/.hermes/state/leadgen_config.json` — источники (enabled/каналы TG),
  `include_re`/`exclude_re` (префильтр), `profile` (текст для LLM),
  `max_candidates_per_run`, `max_llm_per_run`, `active_hours_msk`, `paused`.
- `/root/.hermes/state/leadgen_seen.json` — журнал: `pre` (срезан префильтром),
  `skipped` (LLM: нерелевантно), `proposed` (показан владельцу), `pending`
  (найден ночью, ждёт утра). Повторов нет — ключ `<источник>:<id>`.
- `/root/.hermes/state/leadgen_feedback.json` — `{"bad": [...], "good": [...]}`,
  подмешивается в LLM-промпт каждого прогона, важнее общих правил.

## Реакции владельца на кандидатов (протокол для Гермеса)
Кандидаты приходят с нумерацией, `id <источник>:<номер>`, ссылкой и черновиком.
1. **«N норм»** → отклик владелец шлёт сам по черновику (аккаунты на биржах его);
   Гермес при желании может дописать/подправить черновик.
2. **«N не норм, потому что …»** → добавить причину в `leadgen_feedback.json`
   (`bad`) — тот же приём, что в hh: причина своими словами + название проекта.
   Явную похвалу — в `good`.
3. **Обобщающие пожелания** («не показывай задачи дешевле 10к», «добавь канал X»)
   → правка `leadgen_config.json` (include/exclude, каналы, источники).

## LLM-отбор (правила зашиты в промпт)
Relevant = проект про внедрение ИИ в бизнес: ИИ-агенты/ассистенты, чат-боты,
LLM-интеграции, автоматизация процессов/отчётности, интеграции с CRM
(Битрикс24/amo), n8n/Make/Zapier, парсинг+автоматизация. НЕ брать: чистую
ML/DS-разработку, разработку ПО без ИИ, дизайн/контент/маркетинг, учебные
работы, крипто-казино/сомнительное, разовую мелочь при явном бюджете < 3000 ₽.
Черновик отклика: 2–5 живых предложений, «Здравствуйте!», зацепка за деталь
задачи, один уточняющий вопрос, без штампов и длинных тире, без подписи.

## Cron
Job `leadgen-watch` — каждые 60 минут, no_agent script через обёртку
`/root/.hermes/scripts/leadgen_watch_cron.sh` (stdout → лог; на успехе молчит,
кандидаты уходят в TG напрямую из скрипта; alert в TG только при падении).

## Фазы (план владельца, 2026-07-12)
- **Фаза 1 (эта, live):** мониторинг бирж/тендеров/TG-каналов в review-режиме.
- **Фаза 2:** аккаунты владельца на биржах (Kwork/FL PRO/Workspace-профиль
  подрядчика) → включить Kwork-источник, полуавто-отклики через браузер
  (инфраструктура как `/opt/hh-browser`).
- **Фаза 3 — аутрич-машина (email).** Персональный B2B-аутрич, НЕ массовая
  рассылка (ст. 18 ФЗ «О рекламе»: массовая реклама без согласия — риск штрафа
  ФАС; персональное деловое письмо конкретному ЛПР с гипотезой по его бизнесу —
  рабочая практика). Нужно от владельца: (1) отдельный домен (не основной!),
  (2) выбор 2–3 ниш. Дальше агент: SPF/DKIM/DMARC + прогрев 2–3 недели,
  парсинг компаний ниши (2GIS/каталоги/Rusprofile), изучение сайта компании,
  письмо с конкретной гипотезой автоматизации → review в TG → отправка,
  журнал контактов, автофоллоу-ап через 3–5 дней. Лимит 20–40 писем/день.
- **Фаза 4:** контент (кейс Albery на vc.ru, TenChat) + партнёрка Битрикс24 /
  публикация приложения в Битрикс24.Маркет (стратегический inbound-канал).

## Частые правки
- Сузить/расширить поток → `include_re`/`exclude_re` в конфиге.
- Добавить TG-канал → `sources.telegram.channels` (имя без @; проверить, что
  `https://t.me/s/<канал>` отдаёт превью — не все каналы его разрешают).
- Тон черновиков → `profile` в конфиге + анти-штамп-список в `llm_assess`.
- Разметка сайта-источника поменялась → правь соответствующий `fetch_*` в
  скрипте, проверяй `--list` / `--dry-run --source <имя>`.
